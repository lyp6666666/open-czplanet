# 线上试课闭环开发任务与测试流程

## 1. 开发拆分

### 阶段一：支付前拦截与状态口径

目标：先堵住“教师未支付信息费，学生仍能发起试课/合作/排课”的线上风险。

开发任务：

- 梳理所有入口：
  - 聊天发送
  - 发起合作
  - 修改合作
  - 发起线上约课
  - 试课改期
  - 试课取消
  - 提交正式课表
- 后端增加统一准入校验：
  - `PAYMENT_REQUIRED` 时拒绝聊天、合作、排课
  - `CHAT_CLOSED` 时拒绝普通消息与排课
  - `FINISHED` 时允许聊天但拒绝排课
- 前端根据状态隐藏或禁用入口
- 补充错误提示：
  - 教师端：请先支付信息费
  - 学生端：请等待教师支付信息费

验收重点：

- 未支付信息费前不能通过任何入口发合作或试课
- 直接调接口也会被拒绝

### 阶段二：试课合作提案

目标：将合作提案从“长期课表提案”改为“试课合作提案”。

开发任务：

- 重构合作提案请求字段：
  - `lessonPriceFen` 或兼容 `pricePerHour`
  - `trialStartAt`
  - `trialEndAt`
  - `remark`
  - `clientRequestId`
- 合作提案有效期固定为 12 小时
- 同房间同一时刻只能有一个 `PENDING` 合作提案
- 合作提案系统消息展示：
  - 课时费
  - 试课时间
  - 备注
  - 过期时间
- 同意合作后生成长期课程和试课日程

验收重点：

- 合作提案不再要求“每周频次”
- 合作提案只确认试课，不生成正式固定课表

### 阶段三：双列时间选择器与冲突校验

目标：前端可视化选择试课时间，后端做最终冲突校验。

开发任务：

- 新增双方日程查询接口：
  - 日视图：试课/改期使用
  - 周视图：正式固定课表使用
- 前端实现双列时间选择器：
  - 左列本人
  - 右列对方
  - 点击空白默认 2 小时
  - 支持整体拖动、调整开始、调整结束
- 前端实时冲突提示
- 后端冲突校验：
  - 与本人冲突
  - 与对方冲突
  - 与待确认试课冲突
  - 与正式课表冲突

验收重点：

- 用户可以通过可视化方式找到双方共同空闲时间
- 前端提示不能替代后端校验

### 阶段四：试课改期与取消

目标：试课阶段支持改期和取消，且状态流转一致。

开发任务：

- 我的课程页展示试课阶段
- 新增试课改期提案：
  - 同课程同一时刻只能有一个待处理改期
  - 对方同意后更新试课日程
  - 对方拒绝后保留原时间
- 新增试课取消：
  - 试课开始前任一方可取消
  - 取消后课程回退到 `COMMUNICATING`
  - 取消优先级高于改期，待处理改期自动失效
- 聊天发送对应系统消息

验收重点：

- 试课取消后可以重新发起合作
- 取消和改期并发时，以取消为准

### 阶段五：试课后学生决策与正式课表

目标：试课结束后由学生决定是否继续，并选择正式固定课表。

开发任务：

- 定时任务或扫描任务：
  - 到达试课结束时间，自动进入 `TRIAL_WAIT_STUDENT_DECISION`
- 学生端弹窗：
  - 通过
  - 不通过
- 通过后：
  - 可立即提交正式课表
  - 可稍后选择，进入 24 小时倒计时
- 正式课表：
  - 周视图
  - 可多日多时段
  - 所有时段时长必须一致
  - 仅学生可提交
- 超时未提交：
  - 自动进入 `TRIAL_FAILED`
  - 聊天关闭
  - 教师端展示可退 80% 信息费

验收重点：

- 学生不选择不会让课程无限卡住
- 正式课表不会由教师提交

### 阶段六：提醒、聊天治理与退款提示

目标：补齐提醒和课程状态对聊天权限的影响。

开发任务：

- 生成提醒任务：
  - 截止前 12 小时
  - 截止前 6 小时
  - 截止前 1 小时
- 提醒去重：
  - 学生已提交正式课表后不再提醒
- 邮件提醒：
  - TODO 占位，不阻塞主流程
- 聊天治理：
  - 试课失败/不继续后关闭聊天
  - 正式结课后允许聊天
  - 结课聊天页展示“课程已结课”
- 教师端退款提示：
  - 试课失败后显示可退 80% 信息费

验收重点：

- 提醒不会重复乱发
- 结课后能聊天但不能排课

## 2. 测试流程

### 2.1 单元测试

覆盖范围：

- 合作提案状态流转
- 同房间唯一 `PENDING` 合作提案
- 改期提案唯一性
- 取消优先级高于改期
- 正式课表时长一致校验
- 试课后 24 小时超时失败
- 聊天权限按课程状态切换

建议位置：

- `videoCall-IM-service/src/test/java/...`
- `tutor-appointment-service/src/test/java/...`
- `ai-tutor-web/src/**/*.test.ts`

### 2.2 接口测试

核心接口链路：

1. 创建聊天申请
2. 对方同意
3. 未支付时尝试发合作，预期失败
4. 教师支付信息费
5. 发起试课合作提案
6. 对方同意
7. 查询课程与双方日程，确认试课已生成
8. 发起改期
9. 对方同意/拒绝
10. 取消试课
11. 重新发起合作并完成试课
12. 试课结束后学生选择通过
13. 提交正式课表

### 2.3 E2E 测试

建议新增场景：

- `未支付拦截链路`
  - 申请同意但教师未支付，学生不能发合作/试课
- `试课合作成功链路`
  - 支付 -> 发合作 -> 同意 -> 试课日程生成
- `试课改期链路`
  - 试课已生成 -> 发起改期 -> 同意 -> 日程更新
- `试课取消链路`
  - 试课已生成 -> 取消 -> 回到沟通 -> 可重新发合作
- `试课通过链路`
  - 试课结束 -> 学生通过 -> 提交正式课表 -> 课程 ACTIVE
- `试课超时失败链路`
  - 试课结束 -> 学生未提交正式课表 -> 超时 -> TRIAL_FAILED -> 聊天关闭
- `正式结课聊天开放链路`
  - ACTIVE -> FINISHED -> 聊天可用但排课不可用

### 2.4 回归测试

每阶段至少执行：

- 后端相关模块单测
- 前端相关页面测试
- 手工冒烟一遍主链路

推荐命令：

```bash
sh ./mvnw -pl videoCall-IM-service -am test
sh ./mvnw -pl tutor-appointment-service -am test
cd ai-tutor-web && npm run test
cd ai-tutor-web && npm run typecheck
```

## 3. 上线顺序

1. 先上线支付前拦截，立即修复流程穿透
2. 再上线试课合作提案新语义
3. 再上线试课改期/取消
4. 最后上线试课后正式课表与提醒超时

## 4. 回滚策略

- 支付前拦截可独立回滚，但不建议回滚
- 合作提案字段需兼容旧字段至少一个版本
- 新增表结构只加字段不删字段
- 前端新入口可通过隐藏按钮快速降级
