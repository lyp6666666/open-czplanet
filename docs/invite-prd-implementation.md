# 邀请码与邀请有礼研发落地方案

## 1. 文档信息

- 文档名称：邀请码与邀请有礼研发落地方案
- 文档版本：v1.0
- 面向对象：产品、前端、后端、测试、运营、财务
- 适用范围：`ai-tutor-web`、`tutor-appointment-service`、管理后台、Nacos 配置中心

## 2. 背景与目标

当前平台已形成“注册登录 -> 发布需求 / 申请需求 -> 教师支付信息费 -> 聊天沟通 -> 合作提案通过 -> 退款/课程”等主链路，但缺少围绕老带新的增长机制。

本方案目标如下：

1. 建立每个用户唯一的邀请码机制，支持新用户注册时选填。
2. 建立邀请关系与现有业务链路的绑定，基于真实收入生成返利台账。
3. 新增“邀请有礼”用户入口与页面，提供邀请、查看明细、配置收款微信信息的能力。
4. 将“收藏”从顶部导航迁移到头像下拉菜单，统一个人中心相关入口。
5. 支持通过 Nacos 配置返利比例、结算周期、最低结算门槛、风控开关等核心业务参数。

5. 多级分销。

## 4. 核心业务定义

### 4.1 角色定义

1. 邀请人：分享自己邀请码的存量用户。
2. 被邀请人：注册时填写邀请码的新用户。
3. 邀请关系：邀请人与被邀请人在注册成功时建立的一次性绑定关系。
4. 返利记录：由有效业务事件生成的邀请收益明细。
5. 结算单：按月汇总后的待打款单据。

### 4.2 邀请关系绑定规则

1. 每个用户注册成功时自动生成唯一邀请码。
2. 新用户注册时可填写邀请码，属于选填项。
3. 仅“新注册成功”可绑定邀请码；已有账号登录不允许补绑。
4. 一个账号仅可绑定一次邀请码，绑定后永久不可修改。
5. 自己不能填写自己的邀请码。
6. 机构账号不能作为邀请人，也不能参与返利活动。

### 4.3 返利口径定义

#### 教师邀请返利

推荐口径：

1. 被邀请人账号角色为教师。
2. 被邀请教师参与业务并触发信息费支付成功。
3. 相关合作提案状态变为 `ACCEPTED`，视为成单成立。
4. 对应信息费订单截至结算时未退款、未纠纷冻结。
5. 系统为邀请人生成一条“教师邀请返利记录”。

返利公式：

`返利金额 = 信息费实际支付金额 × 教师返利比例`

#### 学生邀请返利

推荐口径：

1. 被邀请人账号角色为学生。
2. 被邀请学生发布的需求产生了教师真实支付的信息费订单。
3. 截至结算时该订单未退款、未冻结。
4. 系统为邀请人生成一条“学生邀请返利记录”。

返利公式：

`返利金额 = 信息费实际支付金额 × 学生返利比例`

### 4.4 金额口径原则

1. 返利基数统一取“平台实际收到的信息费金额”。
2. 金额统一以“分”为单位存储。
3. 比例变更仅影响新记录，不追溯历史记录。
4. 历史记录必须保留比例快照、金额快照、订单关联快照。

## 7. 状态机设计

### 7.1 邀请关系状态

建议枚举：

1. `ACTIVE`：正常生效
2. `FROZEN`：冻结
3. `INVALID`：失效

### 7.2 返利记录状态

建议枚举：

1. `PENDING`：已生成，待满足结算条件
2. `FROZEN`：因退款、纠纷、风控冻结
3. `SETTLEABLE`：可结算
4. `SETTLEMENT_PENDING`：已进入结算单，待打款
5. `PAID`：已打款
6. `FAILED`：打款失败
7. `REVERSED`：冲正

### 7.3 结算单状态

建议枚举：

1. `CREATED`：已生成
2. `PAYING`：处理中
3. `PAID`：已完成
4. `FAILED`：失败
5. `CANCELED`：取消

## 8. 结算规则

### 8.1 结算周期

1. 按自然月结算。
2. 每月 10 号结算上一个自然月。

示例：

1. 2026 年 4 月 10 日，结算 2026 年 3 月内满足条件的返利。

### 8.2 结算前置条件

1. 返利状态为 `SETTLEABLE`。
2. 邀请人账号状态正常。
3. 已配置微信收款信息。
4. 未命中风控冻结。
5. 汇总金额达到最低打款门槛。

### 8.3 打款失败处理

1. 结算单置为 `FAILED`。
2. 明细记录置为 `FAILED`。
3. 页面给出失败原因或“请补全收款信息”提示。
4. 后续支持重新发起结算或补打款。

## 9. 风控策略

### 9.1 必须拦截

1. 自己邀请自己。
2. 无效邀请码。
3. 非新用户补绑邀请码。
4. 重复绑定不同邀请码。

### 9.2 建议预警

1. 同一设备短时间大量注册并绑定同一邀请码。
2. 同一 IP 短时间大量注册。
3. 多个邀请账号共用同一微信收款信息。
4. 被邀请账号存在异常退款、异常支付行为。

### 9.3 冻结策略

1. 若返利对应订单进入退款或纠纷流程，返利进入 `FROZEN`。
2. 若对应订单最终退款，未结算返利失效；已打款返利生成冲正记录。

## 12. 前后端接口设计

### 12.1 登录注册接口扩展

接口：

`POST /user/loginOrRegister`

请求新增字段：

```json
{
  "phone": "13800138000",
  "code": "1234",
  "userRoleEnum": "STUDENT",
  "inviteCode": "ABC123"
}
```

处理规则：

1. 若命中已有账号登录，不处理邀请码绑定。
2. 若新账号注册成功且填写了邀请码，则建立邀请关系。
3. 若邀请码非法，则注册失败并返回业务错误。

### 14.1 用户域

2. 提供邀请码查询与关系校验服务。
3. 提供返利生成服务与状态流转服务。

### 14.3 事件联动

建议通过领域事件或 MQ 监听以下业务节点：

1. 用户注册成功
2. 信息费支付成功
3. 合作提案通过
4. 退款申请通过 / 退款成功
5. 风控冻结 / 解冻

当前落地说明：

1. IM 服务信息费支付成功后，通过内部接口回调预约服务写入邀请返利事件。
2. 返利明细以 `reward_scene + biz_order_type + biz_order_id` 做幂等控制，确保重试不会重复记账。
3. 月结任务由预约服务定时执行，并在结算单中固化微信收款快照。

1. 无法绑定自己的邀请码。
2. 无法重复绑定邀请码。
3. 退款后返利可冻结或冲正。
4. 配置变更不影响历史明细。
