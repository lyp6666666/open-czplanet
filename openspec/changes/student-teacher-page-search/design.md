## Context

- 当前前端首页在已登录状态下的搜索行为按角色分支：教师跳转到需求列表（tutorJobs）并带 query.q，学生则跳转到发布需求页且不携带关键词；未登录首页搜索入口（HomeHeader）缺少提交行为。
- 前端已有成熟的“教师端需求列表页”工作台布局（搜索 + 筛选 tabs + 左侧列表 + 右侧详情 + 收藏/申请），且学生侧已具备“收藏教师”“查看教师资料卡（含发起申请）”等能力（favoritesTutorsApi / UserCardModal / applicationApi.startChat）。
- 后端已有“热门老师”聚合接口（public/home/hot/tutors），但不支持关键词检索；学生端教师列表页需要关键词筛选与分页支撑，且需要和前端路由 query.q 对齐。

## Goals / Non-Goals

**Goals:**
- 新增学生端教师列表页（/student/tutors），UI 结构与教师端需求列表页一致，展示对象改为教师，并支持关键词 + 基本筛选 + 游标分页。
- 首页搜索按登录态/角色统一跳转策略：学生 → 教师列表页（带 q）；教师 → 需求页（带 q）；未登录 → 弹出“请登录之后再使用搜索功能”提示且不跳转。
- 复用已有“收藏教师”“发起申请（TUTOR 场景）”“用户卡片详情”能力，避免重复造轮子。

**Non-Goals:**
- 不改动既有教师端需求页的筛选字段与接口契约。
- 不引入新的 UI 组件库或全局 Toast 框架（未登录提示采用轻量实现）。
- 不在本次变更内实现复杂的“老师服务广场/服务贴”完整筛选体系（仅满足学生端教师筛选与展示需求）。

## Decisions

- 新增学生端教师列表路由：使用 `/student/tutors`（属于学生端受限路由），由路由守卫保证未登录与角色不匹配时的访问纠正。
- 列表页 UI 复用策略：以教师端需求列表页为参照复制其布局与交互（筛选 tabs、左列表、右详情、加载更多、弹层提示），但数据源与展示字段替换为教师维度（姓名/科目/授课方式/城市/课时费/简介等）。
- 关键词传递约定：统一使用 query `q`（与教师端需求页一致），首页搜索和页内搜索框都写入/读取同一 query，确保刷新/分享链接可复现筛选状态。
- 后端新增接口：新增学生/家长侧“教师列表/搜索”接口，提供游标分页，支持关键词（匹配教师姓名/科目/简介等）与筛选（城市、授课方式等）。响应使用轻量 TutorCardVO（含 userId、avatar、展示名、城市、教育背景、教龄、课时费、标签/亮点）以适配列表展示。
- 详情展示来源：右侧详情通过既有 `/user/card` 获取教师完整档案（TeacherProfile），用于渲染简介等字段；申请与收藏复用现有 favoritesTutorsApi 与 applicationApi.startChat。

## Risks / Trade-offs

- [后端数据模型覆盖不足] 仅基于 teacher_profile/user 做教师列表，服务贴/科目体系可能与未来“服务广场”不一致 → 通过接口参数与 VO 设计保持可扩展（后续可加 subjectId/服务聚合）。
- [未登录提示体验] 现有工程缺少统一 Toast 基建，直接使用轻量提示可能不完全一致 → 先满足需求“弹出提示”，后续如引入全局提示组件可替换实现。
- [筛选字段映射] 教师侧字段（teachingMode/subject）与前端展示选项需要对齐 → 采用字符串枚举（online/offline/both）并在前后端统一归一化处理。
