## Why

当前站内消息在“交换联系方式（QQ/微信/手机号等）”与“岗位立即沟通”场景下容易被简单绕过，导致平台合规与交易闭环风险上升；同时聊天界面信息密度不足、缺少结构化“达成合作”的明确动作，影响转化与后续中介费结算。

## What Changes

- 对聊天消息新增敏感信息识别与展示脱敏：QQ、微信（含 wx/wxid/微信等常见变体）、手机号等命中规则的内容在发送/展示时替换为 `***`，并标记该消息被屏蔽；数据库保留原始文本用于风控与追溯。
- 聊天消息列表改为类似微信的对话样式：左右两侧分别展示发送方头像与姓名；点击对方头像打开名片弹窗，展示对方基础信息。
- 聊天输入框上方新增“发起合作”入口：发起方填写收费标准、上课时间、频次等必要字段后发送一条结构化“合作提案”消息块；接收方可在消息块内选择同意或拒绝，并产生明确状态变更记录，作为后续平台收取中介费的依据。

## Capabilities

### New Capabilities

- `im-message-masking`: 站内 IM 消息的敏感信息识别、展示脱敏与屏蔽标记（存原文 + 展示脱敏 + 命中状态）。
- `im-profile-card`: 聊天头像可点击打开名片弹窗，按用户角色展示基础信息（教师：基本信息/简介；学生：基本信息 + 对应岗位信息）。
- `im-collaboration-proposal`: 在聊天中发起家教合作提案（含收费/时间/频次等字段）、接收方同意/拒绝、状态流转与可追溯记录。

## Impact

- 数据库与数据模型：消息表新增布尔字段（如 `is_masked`）与必要的结构化消息存储字段/扩展字段；需要迁移与兼容历史消息展示。
- 服务端接口：消息发送链路需要在服务端进行敏感信息识别与“展示内容”生成，避免仅依赖前端导致绕过；名片信息与岗位信息需要新增/复用查询接口；合作提案需要新增创建与响应（同意/拒绝）接口与权限校验。
- 前端页面：聊天列表 UI 调整（头像/姓名/气泡布局）、头像点击弹窗、合作提案表单与消息块渲染、同意/拒绝交互及状态刷新。
- 风控与合规：命中规则需可扩展与可配置；展示脱敏不应泄露原文；日志与监控不得输出原始敏感内容。
