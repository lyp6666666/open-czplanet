## Context

当前站内 IM 由 `videoCall-IM-service` 提供 1v1 room 与消息能力（`/chat/room/*`、`/chat/msg`、分页拉取与 SSE 推送）。消息落库在 `message` 表，核心字段为 `content`（文本）、`type`（与 `msgType` 一致，支持 TEXT=1、SYSTEM=8 等）与 `extra`（JSON 扩展）。前端 `ai-tutor-web` 的聊天页以气泡列表渲染消息体 `body`，当前仅区分“自己/对方”的左右对齐，未在消息行内展示头像与姓名；对端用户信息通过 `/user/batch` 获取（包含 `name/avatar/userType`）。

现有系统已支持一种结构化消息卡片：`lesson_request`（课程申请/同意/拒绝），其渲染由聊天页内的 `LessonRequestCard` 完成，交互通过调用 `scheduleApi.respond` 更新状态。

问题与机会：

- 用户可在聊天中交换 QQ/微信/手机号等联系方式，平台难以形成交易闭环且有合规与风控风险。
- 聊天 UI 信息密度不足，缺少微信式“头像 + 姓名 + 气泡”的对话体验。
- “聊定后达成合作”的动作不明确，缺少结构化的合作提案与可追溯的同意/拒绝状态，难以作为后续收取中介费依据。

## Goals / Non-Goals

**Goals:**

- 服务器端对 IM 消息做敏感信息识别与展示脱敏：命中 QQ/微信/手机号等规则的内容在发送/拉取/推送时替换为 `***`，同时数据库保留原文并记录该消息是否命中屏蔽规则。
- 聊天消息列表改为微信式布局：每条消息展示发送方头像与姓名，点击对方头像打开名片弹窗，支持按角色展示基础信息。
- 在聊天输入框上方提供“发起合作”入口：发起方发送结构化“合作提案”消息块，接收方可同意/拒绝并落地状态流转记录，作为收取中介费的依据。

**Non-Goals:**

- 全量内容安全治理（黑名单库、机器审核、图片/语音识别、复杂对抗检测）。
- 将 1v1 room 从“同一对用户唯一”升级为“按岗位/需求贴维度区分多个 room”（本期仅在必要处补充上下文，不改变 room 唯一性规则）。
- 引入新的即时通讯协议或替换现有 SSE 推送通道。

## Decisions

- 敏感信息屏蔽在服务端完成（`videoCall-IM-service`）
  - 选择：在消息保存阶段对用户输入文本做规则匹配，写入 `message.is_masked`；在消息展示阶段（分页拉取、发送返回、SSE 推送）统一输出“脱敏后的展示内容”。
  - 理由：避免仅依赖前端导致绕过；保证历史拉取、实时推送与发送回包三条链路一致。
  - 兼容：对历史消息（`is_masked` 为空/未回填）在展示阶段可再做一次计算以保证老数据也能脱敏展示（不强制回写，回写可作为后续优化）。

- 消息落库存原文，仅新增布尔命中标记（不引入第二份“展示内容”字段）
  - 选择：`message.content` 存原始文本；新增 `message.is_masked`（默认 `false`）。
  - 返回给前端的消息体 `body` 对 TEXT 消息扩展字段：`{ type: 'text', content: '<masked>', masked: true, maskHint?: string }`。
  - 理由：满足“可追溯/风控”的原文留存，同时满足“发出去变成 ***”的展示要求；通过 `body` 扩展避免破坏现有 `ChatMessageResp` 结构。

- 屏蔽规则采用可扩展配置 + 正则匹配的组合
  - 选择：内置基础规则（QQ/微信标识与常见变体、手机号），并预留配置入口（配置中心/DB 表）以便灰度调整；匹配后做“局部替换”为 `***`。
  - 取舍：规则越激进误伤越多；本期优先覆盖典型绕过路径，误伤通过配置快速修正。

- “发起合作”使用 SYSTEM 类型结构化消息卡片，状态由业务表驱动
  - 选择：合作提案消息通过 `msgType=SYSTEM(8)` 发送，`extra` 存放提案 `proposalId` 与快照字段；前端渲染为独立卡片组件，提供同意/拒绝按钮。
  - 同意/拒绝不直接改消息，而是更新“合作提案”业务记录的状态，并回推最新状态（可通过刷新消息/追加一条状态消息/或前端本地更新）。
  - 理由：结构化消息可扩展、可追溯；以业务表作为权威状态，便于后续按提案状态收取中介费与对账。

- 名片弹窗以轻量查询接口支撑，避免在 IM 消息体内冗余塞全量资料
  - 选择：点击头像时由前端请求名片接口获取：教师展示基础资料/简介；学生展示基础资料，并展示“与本会话关联的岗位信息”的摘要。
  - 岗位信息来源策略：优先展示“会话上下文中的需求贴（若存在）”；否则展示该学生最近一条公开/可见需求贴摘要；若仍不可得则只展示学生资料摘要。

## Risks / Trade-offs

- [误伤正常文本（规则过严）] → 规则可配置/可灰度；前端提示“已为你屏蔽敏感信息”，减少困惑；监控命中率与申诉反馈。
- [仅展示脱敏但仍存原文带来合规风险] → 原文仅限必要场景访问（不下发给客户端）；日志/监控严禁输出原文；数据库访问权限收敛。
- [合作提案状态与聊天消息渲染不同步] → 以提案表为权威；同意/拒绝成功后立即刷新对应消息或追加状态消息，保证双方一致可见。
- [岗位上下文缺失导致名片信息不完整] → 支持“最佳努力”回退策略（上下文需求贴 → 最近需求贴 → 仅基础资料）；后续可在“发起沟通”链路补充上下文写入以提升准确性。

## Migration Plan

- 数据库
  - `videoCall-IM-service` 所在库：`message` 表新增 `is_masked`（tinyint/boolean，默认 0），必要时补充索引（按命中率与查询模式评估）。
  - 新增“合作提案”业务表（建议放在与既有 `schedule/appointment` 同域的服务与库中），包含提案内容与状态流转字段。
- 后端
  - `videoCall-IM-service`：在 TEXT 消息保存与展示链路加入脱敏；新增 SYSTEM 消息类型的“合作提案”发送能力与相关校验；补齐对外 DTO 字段（`masked`/`maskHint`）。
  - 资料服务：新增名片查询接口，返回教师/学生基础资料与岗位信息摘要（按可见性做权限控制）。
- 前端
  - 聊天消息行渲染升级（头像/姓名/气泡布局），并接入名片弹窗。
  - 新增“发起合作”入口与提案表单；新增合作提案卡片组件；接入同意/拒绝 API 并处理状态更新。

## Open Questions

- 屏蔽规则的产品边界：仅屏蔽 QQ/微信/手机号，还是同时覆盖邮箱、URL、二维码口令等？
- 被屏蔽后的交互：发送方是否需要明确 toast/弹窗提示，或仅在气泡内展示“已屏蔽”标识？
- 合作提案字段最小集合：除收费标准/上课时间/频次外，是否必须包含授课方式、城市/地址、课程时长、备注等？
- 会话与岗位的上下文关联：是否允许在“发起沟通”时传入 `demandId` 并持久化到 room，以保证名片展示的岗位信息准确？
