## ADDED Requirements

### Requirement: 我的课程必须先展示合作，再展示合作内全部课程

系统 MUST 在“我的课程”中先展示长期合作列表，再在选中的合作详情中展示该合作下的全部课程。

#### Scenario: 打开我的课程页
- **WHEN** 学生或教师进入“我的课程”
- **THEN** 系统 MUST 展示合作列表
- **AND** MUST 支持点击某个合作查看该合作下的全部课程

### Requirement: 系统必须提供合作课程总览页

系统 MUST 为某一合作提供“合作课程总览页”，集中展示全部课节与后续规划。

#### Scenario: 用户查看某个合作的完整课节
- **WHEN** 用户在“我的合作”或聊天页中进入某个合作
- **THEN** 系统 MUST 展示该合作的整体状态
- **AND** MUST 展示最近的一节课、上一节课、已上过的课、正在上的课与后续规划时间

### Requirement: 系统必须提供统一单节课详情弹窗

系统 MUST 在各入口复用同一套“单节课详情弹窗”，而不是跳转到新的大页面。

#### Scenario: 用户点击一节课
- **WHEN** 用户在合作页、合作课程总览页、聊天页或课程表中点击任一课节
- **THEN** 系统 MUST 原地弹出单节课详情
- **AND** MUST 展示这节我要学什么、计划开始/结束时间、当前状态、课后总结和主动作

### Requirement: 聊天页必须展示最近的一节课

系统 MUST 在聊天页展示与当前合作关联的“最近的一节课”。

#### Scenario: 用户进入聊天页
- **WHEN** 当前聊天关联到某一合作
- **THEN** 系统 MUST 展示最近的一节课卡片
- **AND** 卡片 MUST 支持点击后打开单节课详情弹窗

### Requirement: 课程表必须支持 hover 课节预览

系统 MUST 在课程表中支持 hover 课节时的轻量预览，并支持点击查看完整详情。

#### Scenario: 用户浏览课程表
- **WHEN** 鼠标移动到某一节课上
- **THEN** 系统 MUST 展示该课节的轻量预览
- **AND** 点击后 MUST 打开完整单节课详情弹窗

### Requirement: 课程必须按会议事件规则允许进入

系统 MUST 将单节课程视为一个会议事件，并在预约结束时间前始终允许进入课堂。

#### Scenario: 距离预约开始仍有 18 分钟
- **WHEN** 当前时间距离预约开始还有 18 分钟
- **THEN** 系统 MUST 显示 `距开课 18 分钟`
- **AND** MUST 同时允许用户进入课堂

#### Scenario: 提前很久进入课堂
- **WHEN** 当前时间尚未到预约开始，但仍早于预约结束
- **THEN** 系统 MUST 允许进入课堂
- **AND** MUST 提示进入仅代表进入等待态，不立即产生上课数据

#### Scenario: 超过预约结束时间
- **WHEN** 当前时间已超过预约结束时间
- **THEN** 系统 MUST 阻止新的进入课堂动作

### Requirement: 超过预约结束时间后已在课堂中的用户可继续停留

系统 MUST 在超过预约结束时间后阻止新进入，但 MUST NOT 强制踢出已在课堂中的用户。

#### Scenario: 预约已结束但用户已在课堂中
- **WHEN** 课程超过预约结束时间且用户已在课堂中
- **THEN** 系统 MUST 允许该用户继续停留

### Requirement: 结束课程必须依赖实时视频建立记录

系统 MUST 仅在本节课曾成功建立过一次双人实时视频后允许结束课程。

#### Scenario: 尚未建立双人实时视频
- **WHEN** 本节课未检测到双方成功建立实时视频
- **THEN** 系统 MUST 禁用“结束课程”按钮
- **AND** MUST 展示不可结束原因

#### Scenario: 曾建立过一次双人实时视频
- **WHEN** 本节课曾成功建立过一次双人实时视频
- **THEN** 系统 MUST 允许结束课程

### Requirement: 结束课程必须做延时确认

系统 MUST 在用户点击“结束课程”后弹出确认弹窗，并延迟 3 秒后才允许点击确认按钮。

#### Scenario: 打开结束确认弹窗
- **WHEN** 用户点击“结束课程”
- **THEN** 系统 MUST 展示确认弹窗
- **AND** 确认按钮 MUST 在 3 秒后才可点击

### Requirement: 整节课未建立双人实时视频时必须进入异常态

系统 MUST 在整节课未建立双人实时视频且已超过预约结束时间时，将课程置为“待确认未上课 / 异常待处理”。

#### Scenario: 整节课未正常开始
- **WHEN** 当前时间已超过预约结束时间
- **AND** 本节课从未建立双人实时视频
- **THEN** 页面 MUST 展示 `待确认未上课` 或 `异常待处理`
- **AND** MUST 提示 `对方未到场 / 本节未正常开始，等待双方或客服确认`

### Requirement: 异常未上课不得进入正常课后总结流转

系统 MUST 在“待确认未上课 / 异常待处理”状态下阻断正常课后总结流转。

#### Scenario: 课程处于异常未上课
- **WHEN** 课程处于 `待确认未上课 / 异常待处理`
- **THEN** 页面 MUST NOT 展示正常的课后总结生成状态

### Requirement: 课后总结状态必须明确展示

系统 MUST 在课程页明确展示课后总结状态。

#### Scenario: 课后总结生成中
- **WHEN** 课程已结束且课后总结尚未生成完成
- **THEN** 页面 MUST 展示 `课后总结生成中`

#### Scenario: 课后总结已生成
- **WHEN** 课后总结生成成功
- **THEN** 页面 MUST 展示 `课后总结已生成`
- **AND** MUST 提供 `查看课后总结` 入口

#### Scenario: 课后总结生成失败
- **WHEN** 课后总结生成失败
- **THEN** 页面 MUST 展示 `课后总结生成失败`
- **AND** MUST 提供重试入口

### Requirement: 试课通过后的后续动作必须说明清楚

系统 MUST 在合作详情中说明“试课通过 -> 学生确认 -> 支付 -> 后续上课时间确认”的业务闭环。

#### Scenario: 用户查看试课合作详情
- **WHEN** 合作处于试课阶段或试课已结束待确认
- **THEN** 页面 MUST 展示试课通过后的下一步说明

### Requirement: 聊天系统消息必须驱动课程闭环

系统 MUST 在合作通过和试课改期后发送系统消息，引导用户回到“我的课程”。

#### Scenario: 合作通过
- **WHEN** 合作申请通过
- **THEN** 聊天中 MUST 发送一条跳转“我的课程”的系统消息

#### Scenario: 试课改期
- **WHEN** 试课改期完成
- **THEN** 聊天中 MUST 补发一条新的系统消息

### Requirement: 改期和取消必须同步修改日程表

系统 MUST 在课程改期或取消时同步更新双方日程表。

#### Scenario: 试课改期成功
- **WHEN** 某节课改期成功
- **THEN** 系统 MUST 更新双方日程中的对应事件

#### Scenario: 课程取消
- **WHEN** 某节课被取消
- **THEN** 系统 MUST 更新双方日程中的对应事件状态
