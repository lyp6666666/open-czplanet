## Context

现有系统已经具备以下基础能力：

- 长期课程 `Course` 与单节日程 `ScheduleEvent`
- 试课结果提交、正式课约课、聊天房间与系统消息
- 实时课堂 `LiveSession`
- 课后总结结果查询与失败重试接口

缺口不在于单点能力不存在，而在于页面没有把这些能力组织成清晰、可感知、可验证的闭环体验。

## Goals / Non-Goals

**Goals**

- 用户进入“我的课程”后 3 秒内知道“当前状态、当前说明、当前动作”
- 页面以长期合作为一级对象，单节课程为二级对象
- 明确“进入课堂”“结束课程”“课后总结”“异常未上课”的前端规则
- 所有不可执行按钮必须有明确原因
- 试课通过后的业务说明、聊天消息和后续支付/时间确认要形成闭环

**Non-Goals**

- 本期不新增复杂客服处理页面
- 本期不实现“确认未上课”的最终后端 API，只在页面与规格中预留
- 本期不重构课堂页主流程，只保证课程页与课堂入口逻辑口径一致

## Decisions

### 1. “我的课程”先展示合作，再展示合作内全部课程

- 决策：
  - 左侧展示合作列表
  - 右侧展示选中合作的详情、当前课程和合作内全部课程
- 理由：
  - 用户业务认知首先是“我和谁有合作”，其次才是“这个合作下有哪几节课”

### 2. 课程本质是会议事件

- 决策：
  - 进入课堂只代表进入等待态，不立即产生上课数据
  - 在预约结束时间前都允许进入课堂
  - 不因为距离预约开始时间还有 18 分钟、18 小时或更久而阻断进入
- 理由：
  - 与用户真实预期一致，也能兼容“提前等待”和“临近开课快速进入”

### 3. 结束课程必须基于“曾建立过一次双人实时视频”

- 决策：
  - 双方都能点击结束课程
  - 只要本节课曾成功建立过一次双人实时视频，就满足结束条件
  - 不要求双方持续在线
- 理由：
  - 避免因网络抖动、短暂断连导致无法正常结束课程

### 4. 结束课程必须做强确认

- 决策：
  - 点击结束课程后弹出确认弹窗
  - 确认按钮等待 3 秒后才可点击
  - 暂不支持误点回退
- 理由：
  - 对“结束课程并启动课后生成”这一不可逆操作做必要保护

### 5. 未建立双人实时视频的超时课程进入异常态

- 决策：
  - 到预约结束时间前，课程仍保持“未开始”
  - 到预约结束时间后，如整节课未建立双人实时视频，则进入“待确认未上课 / 异常待处理”
  - 此状态下不走正常课后总结流程
- 理由：
  - 避免把爽约、网络异常、临时改期等复杂场景误判为正常上课完成

### 6. 课后总结状态必须前台可感知

- 决策：
  - 课后总结状态统一展示为：生成中、已生成、生成失败
  - 失败时在课程页提供“重试生成”
- 理由：
  - 避免用户点击空页面，也降低“到底是不是没生成”的困惑

## Page Model

### 页面结构

1. 顶部说明区
2. 合作列表区
3. 合作详情头部
4. 当前课程卡片
5. 合作内全部课程
6. 课后结果区
7. 流程说明区

### 当前课程卡片必须展示

1. 课程时间
2. 当前课节状态
3. 倒计时或当前状态说明
4. 进入规则说明
5. 主按钮：进入课堂
6. 次按钮：查看课程详情、结束课程
7. 不可结束原因

### 合作内全部课程列表必须展示

1. 课节类型：试课 / 正式课
2. 时间
3. 状态
4. 简要说明

## Derived UI Rules

### 进入课堂

- `canEnterClassroom = lesson.status in [ACCEPTED, COMPLETED] && now < lesson.endAt`

展示：

- 若可进入，显示 `进入课堂`
- 文案提示：
  - `课程在预约结束前都可以进入；现在也能进入课堂等待对方`

### 结束课程

- `canEndClassroom = live.sessionId exists && (live.status == IN_PROGRESS || live.actualStartAt exists) && live.status != ENDED`

展示：

- 若不可结束，按钮置灰并展示：
  - `未检测到双方成功建立实时视频，暂不可结束课程`

### 异常未上课

- `showAbnormalAttendanceConfirm = lesson.status == ACCEPTED && now >= lesson.endAt && !hasLiveStarted`

展示：

- 状态标签：`待确认未上课`
- 说明文案：
  - `对方未到场 / 本节未正常开始，等待双方或客服确认`
- 课后结果不进入正常生成态

### 课后总结

- `READY -> 课后总结已生成`
- `FAILED -> 课后总结生成失败`
- `lesson ended && !READY && !FAILED -> 课后总结生成中`
- `abnormal pending confirm -> 未走正常课后流程`

2. `LivePreparePage` / `prepare` 接口最终对齐“预约结束前均可进入”
3. 聊天系统消息补齐：
   - 合作通过
   - 试课改期
   - 课后总结已生成
4. 改期和取消必须同步更新双方日程
