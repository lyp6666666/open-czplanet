# 设计：微信小程序端重建流程

## 当前梳理结果

### 已存在但不可直接视为可用

- `ai-tutor-miniprogram` 是 Uni-app + Vue 3 + TypeScript 项目，已接入 Pinia 与 uView Plus。
- 当前已有页面：首页、我的、聊天列表/聊天室、发需求、我的需求、导师详情、需求详情、教师入驻、邮箱设置。
- 当前已有封装：`request.ts`, `user.ts`, `jobs.ts`, `chat.ts`, `tutor.ts`。
- 当前 `npm run type-check` 通过，说明 TypeScript 层面未阻塞，但这不代表业务可用。

### 主要问题

- 角色与准入不可靠：`currentRole`、`userType`、`teacherProfile`、教师认证状态之间缺少统一口径。
- 业务主链路过期：页面仍偏“直接聊天/直接找人”，没有完整收口到当前 Web 端的“申请 -> 通过 -> 教师支付信息费 -> 解锁聊天 -> 试课合作”。
- 支付缺失：小程序端没有可用的信息费支付、订单状态轮询、支付后解锁和失败恢复。
- IM 能力不足：缺少实时/同步事件、已读回执、系统消息卡片、申请/支付/合作状态卡片。
- 课程履约缺失：试课合作、改期、取消、学生试课结果、正式课表、我的课程没有形成小程序入口。
- 接口类型缺失：大量 `any`，页面直接拼接后端字段，后续维护风险高。
- 页面体验不成体系：表单、空状态、错误态、骨架屏、登录拦截、返回路径不统一。

## 架构决策

### 1. 保留 Uni-app，但重建业务层

继续使用现有 Uni-app 项目和微信小程序构建链路，避免重新建仓。业务层按领域重建：

- `src/api`: 每个后端领域一个模块，返回值使用 TypeScript 类型。
- `src/stores`: 只存跨页面状态，如认证、角色、未读数、支付轮询中的订单。
- `src/pages`: 页面只负责交互编排，不直接散落复杂字段转换。
- `src/components`: 提炼业务卡片、列表项、表单控件、状态视图。
- `src/utils`: 请求、图片、日期、金额、状态文案、轮询、权限判断。

### 2. 以当前 Web 端业务为小程序基线

小程序不是独立业务分支。凡涉及状态机，必须对齐当前 Web 端和后端：

- 找家教主链路：申请 -> 对方通过 -> 教师支付信息费 -> 聊天解锁。
- 线上合作主链路：聊天解锁 -> 试课合作提案 -> 试课日程 -> 试课后学生决策 -> 正式课表。
- 线下模式保留信息费与联系方式解锁，后续履约不进入平台课程体系。

### 3. 登录后统一刷新用户上下文

任何登录方式成功后都必须执行同一套初始化：

1. 保存 JWT。
2. 请求 `/user/me` 获取完整资料。
3. 归一化用户身份：是否学生、是否教师、教师资料是否完整、教师认证状态。
4. 决定默认角色：有明确上次角色则保留；否则学生优先，已完成教师入驻的用户可切到教师。
5. 刷新未读申请数、未读消息数和必要配置。
6. 根据登录前来源回跳原页面或进入首页。

### 4. 小程序端实时策略

微信小程序不稳定支持 Web EventSource。首期采用“事件同步 + 短轮询”：

- 聊天列表每 10-15 秒刷新会话摘要和未读。
- 聊天室进入后拉取历史消息，并按 `lastEventId` 调用 `/chat/events/sync` 增量同步。
- 发送消息后立即本地插入 pending 消息，成功后替换，失败可重试。
- 后续可扩展 WebSocket，但首期不强依赖新后端。

## 页面与路由规划

### TabBar

- 首页：按当前角色展示学生版/教师版首页。
- 消息：申请提醒、聊天列表、未读入口。
- 我的：登录、资料、角色切换、课程、设置。

### 分包建议

- 主包：登录、首页、消息列表、我的。
- `student` 分包：导师列表/详情、发布需求、我的需求、申请详情。
- `tutor` 分包：需求列表/详情、入驻、认证、教师资料。
- `chat` 分包：聊天室、申请中心、支付页、合作提案。
- `course` 分包：我的课程、课程详情、课表、试课结果、退款申请。

## 详细流程

### 流程 1：启动与运行配置

1. App 启动读取环境配置：`VITE_API_BASE_URL`、本地覆盖配置、构建环境。
2. 校验 API Base URL，不允许生产包默认指向 `localhost`。
3. 初始化请求层：统一 Bearer Token、401 处理、错误 toast、请求 loading、幂等请求标识。
4. 读取本地 token 和角色缓存。
5. 若存在 token，调用 `/user/me` 刷新资料。
6. 若 token 失效，清理 token、用户资料、角色缓存，并保持在可浏览的游客态。
7. 首页拉取公共配置：热门词、Banner、科目树、定位或城市默认值。

### 流程 2：登录与角色初始化

1. 用户进入“我的”或触发受保护动作。
2. 可选择手机号验证码登录或微信登录。
3. 手机号登录：
   1. 调用 `/user/sendcode`。
   2. 输入验证码后调用 `/user/loginOrRegister`。
   3. 传入期望角色，但后端返回身份才是准入依据。
4. 微信登录：
   1. 调用 `uni.login({ provider: 'weixin' })` 获取 code。
   2. 调用 `/user/wechatLogin` 换取 JWT。
   3. Dev/Mock 模式下允许使用 mock code，但必须有显式环境开关。
5. 登录成功后执行用户上下文初始化。
6. 若登录前有 redirect URL，回跳原动作；否则进入当前角色首页。
7. 登录失败保留表单内容，展示后端错误，不吞掉真实错误。

### 流程 3：角色切换与教师准入

1. “我的”页展示当前角色和可切换身份。
2. 切到学生版：直接切换并回到首页。
3. 切到教师版：
   1. 若无教师资料，引导到教师入驻。
   2. 若资料未提交完整，继续入驻草稿。
   3. 若认证待审核，进入教师状态页。
   4. 若被拒绝，展示原因并允许重新提交。
   5. 若通过，切换到教师首页。
4. 所有教师专属页面进入前都要调用统一守卫。
5. 所有学生专属写操作也要调用统一守卫，避免仅靠页面按钮隐藏。

### 流程 4：学生/家长找老师

1. 学生首页展示搜索框、科目/年级/授课形式筛选、推荐老师。
2. 调用 `/api/v1/parent/tutors/page` 获取教师列表。
3. 列表项展示头像、姓名、城市、学校、科目、课时费、简介、收藏状态。
4. 点击进入教师详情。
5. 详情页拉取教师资料，展示服务范围、履历、认证状态、可授课方式。
6. 用户点击收藏：
   1. 未登录则跳登录。
   2. 已登录调用 `/api/v1/parent/favorites/tutors/{tutorId}`。
7. 用户点击“发起沟通申请”：
   1. 未登录则跳登录并回到详情页。
   2. 弹出申请表，预填默认问候语和关联教师上下文。
   3. 调用 `/chat/application` 或 `/chat/application/start-chat` 创建申请。
   4. 成功后跳转申请详情或消息页。

### 流程 5：学生发布与管理需求

1. 进入发布需求页前必须登录且角色为学生。
2. 表单字段至少包括：科目、阶段/年级、授课形式、城市/地址、预算、频次、可上课时间、教师要求、需求描述。
3. 授课形式选择后决定后续闭环：
   1. 线上：后续可进入试课与正式课程流程。
   2. 线下：后续以信息费、联系方式解锁和线下履约为主。
4. 前端做必填校验、金额范围校验、线上/线下字段差异校验。
5. 提交调用 `/api/v1/parent/jobs`。
6. 我的需求调用 `/api/v1/parent/jobs/mine`，支持状态筛选。
7. 需求详情调用 `/api/v1/parent/jobs/{id}`。
8. 支持编辑时调用 `PUT /api/v1/parent/jobs/{id}`，已进入申请/合作阶段的字段按后端规则锁定。
9. 需求详情展示收到的申请入口，学生可进入申请中心处理。

### 流程 6：教师入驻与资料认证

1. 用户切换教师版但未完成入驻时进入入驻向导。
2. 步骤一：基础资料，头像、姓名、城市、个人简介。
3. 步骤二：教学信息，科目、年级、授课方式、课时费、可授课时间。
4. 步骤三：学历/认证资料，学校、学历、证书图片。
5. 图片上传统一调用 `/api/v1/assets/upload`，显示上传进度、失败重试和可删除。
6. 资料保存调用 `/user/updateUserInfo` 或对应教师资料接口。
7. 学历认证调用 `/teacher/verification/education/submit`。
8. 实名认证调用 `/teacher/verification/realname/submit`。
9. 提交后进入教师状态页，展示待审核、通过、拒绝及拒绝原因。
10. 审核通过后允许进入教师首页和需求广场。

### 流程 7：教师浏览需求与发起申请

1. 教师首页展示需求广场，调用 `/api/v1/parent/jobs/feed`。
2. 支持关键词、授课形式、城市、科目、预算筛选。
3. 点击需求进入教师视角详情 `/api/v1/parent/jobs/{id}/view`。
4. 详情展示需求、发布者摘要、授课形式、预算、时间和教师要求。
5. 点击收藏调用 `/api/v1/tutor/favorites/demands/{demandId}`。
6. 点击“申请沟通”：
   1. 校验教师资料和认证状态。
   2. 弹出申请内容表单。
   3. 调用 `/chat/application` 创建申请。
   4. 创建成功后进入发出的申请详情。

### 流程 8：申请中心

1. 消息 Tab 顶部展示“申请”和“聊天”两个入口。
2. 发出的申请调用 `/chat/application/sent/page`。
3. 收到的申请调用 `/chat/application/received/page`。
4. 未读数调用 `/chat/application/unread`。
5. 申请详情调用 `/chat/application/{applicationId}`。
6. 接收方可通过或拒绝：
   1. 通过/拒绝调用 `/chat/application/{applicationId}/decision`。
   2. 成功后刷新申请详情和列表状态。
7. 通过后不直接开放聊天：
   1. 若当前用户为教师且需要支付，展示“去支付信息费”。
   2. 若当前用户为学生，展示“等待教师支付后开启聊天”。
8. 支付完成后调用 `/chat/application/{applicationId}/enter-chat` 进入聊天室。

### 流程 9：信息费支付与聊天解锁

1. 教师在申请通过后进入支付页。
2. 支付页基于申请或中介费订单创建支付上下文。
3. 调用 `/payment/prepay` 获取订单号、渠道、金额、支付参数。
4. 小程序首期优先支持微信 JSAPI 支付；若后端仅返回扫码链接，则展示“开发环境扫码/Mock 支付”兜底，不作为生产方案。
5. 调用 `uni.requestPayment`。
6. 支付发起后轮询 `/payment/orders/{orderNo}`。
7. 支付成功后刷新申请详情，确认聊天权限进入 `CHAT_ENABLED`。
8. 支付失败、取消、超时都必须保留订单状态和重试入口。
9. Dev 环境可调用 `/payment/dev/orders/{orderNo}/mock-success`，但生产包必须禁用。

### 流程 10：聊天列表与聊天室

1. 聊天列表调用 `/chat/room/page`，展示头像、名称、最后消息、未读数、业务状态。
2. 进入聊天室后调用 `/chat/public/msg/page` 拉取历史消息。
3. 聊天室启动增量同步：
   1. 记录最新事件 ID 或最新消息 ID。
   2. 定时调用 `/chat/events/sync`。
   3. 合并新消息、已读、撤回、系统卡片、状态变更。
4. 发送文本调用 `/chat/msg`。
5. 进入聊天室后调用 `/chat/read/ack` 标记已读。
6. 消息展示必须支持：
   - 普通文本。
   - 图片消息。
   - 申请状态卡片。
   - 信息费支付卡片。
   - 合作提案卡片。
   - 试课/正式课表系统消息。
   - 退款/联系方式解锁状态。
7. 当 `chatAccessStatus=PAYMENT_REQUIRED` 时，前端阻断普通输入，并展示支付/等待支付引导。
8. 后端仍必须兜底阻断，前端不能作为唯一门禁。

### 流程 11：试课合作与课程

1. 聊天解锁后，双方可发起试课合作提案。
2. 发起提案调用 `/chat/collaboration/proposal`，字段包括课时费、试课日期、开始时间、结束时间、备注。
3. 发起前查询双方当日可用时间 `/api/v1/schedule/availability/day`。
4. 前端展示双列时间表，用户只能选择不冲突时段。
5. 接收方通过或拒绝调用 `/chat/collaboration/proposal/{proposalId}/response`。
6. 通过后我的课程出现试课课程，课程列表调用 `/courses/my`。
7. 课程详情调用 `/courses/{courseId}`。
8. 试课可改期、取消，相关动作优先从课程详情进入，并通过聊天系统消息通知对方。
9. 试课结束后学生提交结果：
   1. 通过：进入正式课表选择。
   2. 不通过：填写原因和必要凭证，进入失败/退款相关流程。
10. 正式课表提交调用 `/api/v1/schedule/courses/{courseId}/weekly-schedule`。
11. 正式课表提交成功后课程进入正式授课阶段。

### 流程 12：我的课程与退款入口

1. “我的”页提供我的课程入口。
2. 课程列表区分：待确认试课、已确认试课、待学生决策、待补正式课表、正式课、已结束。
3. 课程详情展示双方信息、授课形式、试课/正式课表、支付状态、下一步动作。
4. 试课不通过后，教师可按规则申请信息费退款。
5. 退款状态调用 `/chat/refund/state`，申请调用 `/chat/refund/apply` 或课程退款接口。
6. 退款材料上传复用资源上传组件。

### 流程 13：账号与设置

1. “我的”页展示头像、昵称、手机号、当前角色、教师状态。
2. 邮箱提醒调用 `/user/email`、`/user/email/code`、`/user/email/verify`、`DELETE /user/email/summary`。
3. 支持退出登录，退出后清理 token、用户资料、角色、未读数和支付轮询。
4. 支持 API 地址开发配置，但生产包隐藏该入口。

## 验收策略

### 最低技术门禁

- `cd ai-tutor-miniprogram && npm run type-check`
- `cd ai-tutor-miniprogram && npm run build:mp-weixin`
- 生产构建检查：API Base URL 不得为 `localhost`。

### 核心冒烟链路

1. 游客打开首页，能浏览公共推荐，点击受保护动作跳登录。
2. 学生手机号登录，发布需求，查看我的需求。
3. 教师登录，完成入驻状态判断，浏览需求并提交申请。
4. 学生收到申请并通过。
5. 教师支付信息费，支付成功后进入聊天。
6. 双方发送消息，未读/已读状态正确。
7. 聊天中发起试课合作，对方同意后我的课程出现试课。
8. 学生试课后选择通过并提交正式课表。
9. 失败/取消/支付超时都能回到可恢复状态。
