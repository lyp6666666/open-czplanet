# 小程序现有页面审计与处置建议

> 结论：当前小程序可作为代码素材库，但不能作为业务基线继续补丁式开发。首期应按 `design.md` 的流程重建页面、接口类型和状态守卫。

## 页面处置表

| 现有页面 | 当前问题 | 处置 |
| --- | --- | --- |
| `pages/home/index.vue` | 未登录 `onShow` 直接跳转“我的”，破坏游客浏览老师/需求的首屏体验；只按本地 `currentRole` 切组件，无登录上下文初始化。 | 重写。首页应允许游客浏览公共推荐，受保护动作再跳登录。 |
| `pages/home/components/ParentHome.vue` | 已有教师列表雏形，但筛选、收藏、申请闭环、游客态不足；字段仍大量 `any`。 | 保留展示思路，重写数据层和交互。 |
| `pages/home/components/TutorHome.vue` | 已有需求广场雏形，但教师准入、收藏、申请中心联动不足。 | 保留列表卡片思路，重写守卫和申请入口。 |
| `pages/me/index.vue` | 登录后只展示少量入口；角色切换依赖不可靠的 `tutorStatus`；缺少我的课程、申请中心、开发配置、完整退出清理。 | 重写。 |
| `components/LoginCard.vue` | 可复用部分视觉和验证码表单；需要接入统一登录回跳、Mock 环境限制和错误态。 | 局部保留，重构事件契约。 |
| `pages/tutor/detail.vue` | 教师详情展示较完整，但同时保留“预约课程”旧流程；申请成功后可能直接进聊天室，不符合当前“申请 -> 通过 -> 支付 -> 聊天”主链路。 | 重写业务动作，保留资料展示样式素材。 |
| `pages/job/detail.vue` | 教师视角需求详情可展示字段，但“发起沟通”直接调用 `/chat/room` 创建房间，绕过申请和支付门禁。 | 重写沟通入口。 |
| `pages/post/index.vue` | 表单是英文 Demo 风格；字段、校验、线上/线下授课形式、可上课时间都不完整。 | 重写。 |
| `pages/my-jobs/index.vue` | 简单列表，使用 `res.items`，但现有其他分页多为 `res.list`；缺少状态、编辑、申请入口和空/错态。 | 重写。 |
| `pages/chat/list.vue` | 只有房间列表轮询；缺少申请入口、未读申请、业务状态、已读同步。 | 重写为消息工作台。 |
| `pages/chat/room.vue` | 有部分系统卡片雏形，但仍按 3 秒全量轮询；输入区不检查 `chatAccessStatus`；支付走旧 `/payment/create`；文案混用英文。 | 重写，保留卡片识别经验。 |
| `pages/tutor/onboarding/index.vue` | 有三步入驻雏形和上传能力；教师状态推断粗糙，认证提交与资料保存缺少明确状态流。 | 保留上传和步骤结构，重写状态与字段。 |
| `pages/tutor/status.vue` | 状态页可复用，但状态来源只靠路由参数，未从后端刷新。 | 局部保留，改为真实状态页。 |
| `pages/account/email.vue` | 邮箱设置相对独立，接口与现有后端匹配度较高。 | 保留并做样式/错误态统一。 |

## 需要优先删除或废弃的旧行为

- 未登录打开首页直接跳“我的”。
- 教师需求详情直接 `/chat/room` 创建聊天室。
- 教师详情“预约课程 -> 确认并支付”的旧预约支付入口。
- 聊天页使用 `/payment/create` 作为信息费支付主入口。
- `uni.login` 失败后无条件尝试 mock code。
- 通过 `teacherProfile` 是否存在直接判定教师审核通过。
- 页面内散落英文文案、`any` 类型、硬编码状态和接口字段。

## 可复用资产

- Uni-app 工程、Vite 构建、Pinia、uView Plus 接入。
- TabBar 图标、默认头像、基础样式变量。
- `request.ts` 的基础 Bearer Token 注入思路。
- 教师详情、需求详情、入驻页的部分视觉结构。
- 邮箱设置页的主体流程。
