## Context

当前仓库已经具备以下基础能力，可作为本次方案的复用底座：

- 聊天申请、教师支付信息费、聊天解锁已有基础链路
- `videoCall-IM-service` 已承接房间、消息、系统消息、合作提案与课程视图基础能力
- `tutor-appointment-service` 已承接日程页、课程日历事件、课程安排相关接口
- `ai-tutor-web` 已有聊天页、课程页、日程页、结构化消息卡片等基础页面
- 现有 `openspec` 已有合作提案、试课退款与线上课程相关规格，可作为本次变更的修改基础

本次的重点不是增加单点功能，而是统一线上链路中“支付解锁聊天 -> 试课合作提案 -> 试课履约 -> 学生确认正式课表 -> 正式授课”的完整状态机。

## Goals / Non-Goals

**Goals**

- 教师支付信息费前，系统 MUST 在前后端双重阻断聊天、合作和排课能力
- 合作提案 MUST 只承载试课确认信息，不再直接承载正式长期课表
- 试课时间 MUST 通过双方可视化日程进行选择，并进行前后端双重冲突校验
- 试课改期与取消 MUST 有明确入口、明确状态、明确系统消息反馈
- 试课结束后学生 MUST 做继续/不继续决策；若继续，正式课表 MUST 在 24 小时内补齐
- 聊天权限 MUST 与试课状态、课程状态统一治理
- 核心动作 MUST 幂等、防重、可追溯

**Non-Goals**

- 不支持教师发起建议正式课表
- 不支持多人课、小班课、拼课
- 不支持非北京时间
- 不在本期实现完整邮件提醒，只预留 TODO
- 不在本期实现复杂自动分账或财务核销能力

## Decisions

### 1. 时区统一按北京时间处理

- 决策：
  - 所有试课时间、正式课表、提醒时间、课程展示、倒计时计算 MUST 使用北京时间（GMT+8）
- 理由：
  - 当前产品与用户均以国内使用场景为主，避免跨时区复杂度

### 2. 合作提案重定义为试课合作提案

- 决策：
  - 合作提案 MUST 只包含课时费、试课日期、试课起止时间、备注
  - 正式固定课表 MUST 在试课结束后由学生单独确认
- 理由：
  - 将试课确认与长期授课确认拆开，避免语义和状态混乱

### 3. 试课时间选择必须可视化

- 决策：
  - 发起合作与试课改期都 MUST 使用可视化双列时间选择器
  - 左列展示本人当日日程，右列展示对方当日日程
  - 默认点击空白处生成 2 小时试课区块，且试课时长可调
- 理由：
  - 让双方基于可视化空闲时间协同排课，减少人工沟通成本

### 4. 正式固定课表必须由学生确认

- 决策：
  - 仅学生可提交正式固定课表
  - 正式固定课表可包含一周内多日不同时间段
  - 同一课程下所有固定时段的时长 MUST 一致
- 理由：
  - 符合用户角色定位，也简化后续课节生成与计费逻辑

### 5. 试课结束后自动进入待学生决策

- 决策：
  - 试课结束时间自然到达后，系统 MUST 自动切换到待学生决策状态
  - 同时保留人工兜底能力，以便异常场景修复
- 理由：
  - 避免依赖人工操作推进主流程

### 6. 改期并发与取消优先级明确

- 决策：
  - 同一时刻同一课程只能存在一个待处理试课改期提案
  - 试课取消优先级高于改期；取消后未决改期提案 MUST 自动失效
- 理由：
  - 避免状态竞争导致用户看到不一致信息

### 7. 聊天权限由课程状态驱动

- 决策：
  - 试课不通过或学生不继续后，聊天 MUST 关闭
  - 正式结课后，聊天 MUST 继续开放
  - 结课后聊天页 MUST 展示“课程已结课”标签，且 MUST NOT 再允许排课相关动作
- 理由：
  - 聊天不仅是 IM 能力，也是履约与售后边界的一部分

## State Machines

### 1. 聊天申请状态

- `PENDING`
- `ACCEPTED`
- `REJECTED`

### 2. 聊天准入状态

- `NONE`
- `PAYMENT_REQUIRED`
- `CHAT_ENABLED`
- `CHAT_CLOSED`
- `CHAT_ENABLED_FINISHED`

约束：

- `PAYMENT_REQUIRED` 时 MUST NOT 聊天、发合作、改期、排课
- `CHAT_CLOSED` 时 MUST NOT 发送普通聊天消息
- `CHAT_ENABLED_FINISHED` 时 MAY 聊天，但 MUST NOT 排课

### 3. 提案状态

合作提案与改期提案共用以下状态集：

- `PENDING`
- `ACCEPTED`
- `REJECTED`
- `CANCELED`
- `EXPIRED`
- `INVALIDATED`

### 4. 长期课程状态

- `COMMUNICATING`
- `TRIAL_PENDING_CONFIRM`
- `TRIAL_SCHEDULED`
- `TRIAL_RESCHEDULE_PENDING`
- `TRIAL_IN_PROGRESS`
- `TRIAL_WAIT_STUDENT_DECISION`
- `TRIAL_WAIT_WEEKLY_SCHEDULE`
- `ACTIVE`
- `TRIAL_FAILED`
- `FINISHED`

## Flows

### Flow A: 申请通过到聊天解锁

1. 一方发起聊天申请
2. 对方同意
3. 系统将聊天准入状态置为 `PAYMENT_REQUIRED`
4. 教师完成信息费支付
5. 系统将聊天准入状态置为 `CHAT_ENABLED`
6. 双方可进入聊天

### Flow B: 发起试课合作提案

1. 用户在聊天页点击“发起合作”
2. 系统校验当前聊天已解锁且无待处理合作提案
3. 用户选择试课日期
4. 系统拉取双方当日日程
5. 用户在双列时间选择器中选择试课时间
6. 前端实时校验冲突
7. 用户提交合作提案
8. 后端再次校验时间冲突、提案唯一性与幂等
9. 系统写入合作提案并发送结构化系统消息
10. 课程阶段进入 `TRIAL_PENDING_CONFIRM`

### Flow C: 同意合作提案并生成试课

1. 接收方同意合作提案
2. 系统将提案状态置为 `ACCEPTED`
3. 创建或更新长期课程
4. 课程阶段置为 `TRIAL_SCHEDULED`
5. 生成双方试课日程，标记为“试课”
6. 聊天发送“试课已安排”系统消息

### Flow D: 试课改期

1. 用户从“我的课程”进入试课详情
2. 点击“改期”
3. 进入双列时间选择器并选择新时间
4. 提交改期提案
5. 系统发送改期系统消息并等待对方确认
6. 对方同意后更新试课时间并回到 `TRIAL_SCHEDULED`
7. 对方拒绝后保留原试课时间并发系统消息

### Flow E: 试课取消

1. 任一方在试课开始前点击取消
2. 系统校验当前时间早于试课开始
3. 课程阶段回退到 `COMMUNICATING`
4. 当前试课日程作废
5. 待处理改期提案若存在则置为 `INVALIDATED`
6. 聊天发送“试课已取消，已回到沟通阶段”系统消息

### Flow F: 试课结束后的学生决策

1. 试课结束时间到达
2. 系统自动将课程阶段切为 `TRIAL_WAIT_STUDENT_DECISION`
3. 学生收到弹窗或课程待办
4. 学生选择：
   - `PASS`
   - `FAIL`
5. 若 `FAIL`
   - 课程置为 `TRIAL_FAILED`
   - 聊天关闭
6. 若 `PASS` 且已提交正式课表
   - 课程置为 `ACTIVE`
7. 若 `PASS` 但暂未提交正式课表
   - 课程置为 `TRIAL_WAIT_WEEKLY_SCHEDULE`
   - 写入 24 小时截止时间
   - 生成提醒任务

### Flow G: 正式课表选择与超时

1. 学生进入正式课表选择页
2. 在一周视图中选择一个或多个固定时段
3. 前端校验时长一致、时间冲突与完整性
4. 后端再次校验并保存规则
5. 保存成功后课程置为 `ACTIVE`
6. 取消剩余提醒任务
7. 聊天发送“正式课表已确认”系统消息
8. 若超时仍未提交，则自动置为 `TRIAL_FAILED`

## API and Backend Changes

- 合作提案接口改造为试课合作提案语义
- 新增双方当日日程查询接口
- 新增试课改期提案接口
- 新增试课取消接口
- 新增学生试课决策接口
- 新增正式课表提交接口
- 扩展我的课程与课程详情接口返回字段
- 扩展系统消息 `bizType`

## Frontend Changes

- 聊天页：
  - 发起合作入口
  - 系统消息扩展
  - 已结课标签
- 合作弹窗：
  - 试课日期 + 双列时间选择器 + 备注
- 我的课程页：
  - 待确认试课 / 已确认试课 / 正式每周课 三层展示
  - 改期、取消、待确认正式课表入口
- 正式课表页：
  - 周视图
  - 多时间段选择
  - 时长一致校验
- 学生端试课结束弹窗：
  - 通过 / 不通过
  - 稍后选择正式课表

## Risks

- 双列时间选择器的拖拽与边界处理复杂，需重点验证移动端和桌面端
- 冲突校验需要统一合并试课、正式课、待确认提案等多类占用源
- 超时任务与人工操作之间存在并发竞争，需通过乐观锁、幂等键和状态前置校验避免脏写
- 旧合作提案模型与新提案模型需要兼容迁移

## Rollout

建议按以下模块拆分实施：

1. 支付前聊天/合作/排课拦截
2. 试课合作提案与双列时间选择器
3. 试课改期与取消
4. 试课结束决策与正式课表
5. 提醒与超时失败
6. 聊天权限治理与结课标签
