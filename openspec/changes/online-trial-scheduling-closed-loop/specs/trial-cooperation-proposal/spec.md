## ADDED Requirements

### Requirement: 合作提案必须重定义为试课合作提案

系统 MUST 将线上合作提案建模为“试课合作提案”，其确认内容 MUST 包含课时费、试课日期、试课时间和备注，而 MUST NOT 直接承载正式每周固定课表。

#### Scenario: 发起合作提案时填写试课信息
- **WHEN** 任一方在已解锁聊天中发起合作提案
- **THEN** 系统 MUST 要求填写课时费、试课日期、试课开始时间、试课结束时间和备注

### Requirement: 合作提案需要通过双方可视化日程选择试课时间

系统 MUST 提供按北京时间展示的双方双列时间选择器，并允许用户在选择试课日期后查看本人和对方的当日日程占用情况。

#### Scenario: 选择试课日期后查看双方日程
- **WHEN** 用户在合作提案中选择某个试课日期
- **THEN** 系统 MUST 展示该日期下双方的双列时间视图

#### Scenario: 点击空白区域默认生成2小时试课块
- **WHEN** 用户在自己的时间列中点击空白区域
- **THEN** 系统 MUST 默认生成一段时长为 2 小时的试课时间块

#### Scenario: 试课块支持调整
- **WHEN** 用户拖动试课时间块
- **THEN** 系统 MUST 支持整体移动、调整开始时间和调整结束时间
- **AND** 试课时长 SHOULD 默认为 2 小时且可调

### Requirement: 合作提案提交前必须完成冲突校验

系统 MUST 在前后端双重校验试课时间与双方日程冲突。

#### Scenario: 前端发现本人冲突
- **WHEN** 用户选择的试课时间与本人已有日程冲突
- **THEN** 前端 MUST 实时提示冲突并阻止提交

#### Scenario: 前端发现对方冲突
- **WHEN** 用户选择的试课时间与对方已有日程冲突
- **THEN** 前端 MUST 实时提示冲突并阻止提交

#### Scenario: 后端二次校验冲突
- **WHEN** 用户提交合作提案
- **THEN** 后端 MUST 再次校验双方日程冲突
- **AND** 若存在冲突则 MUST 拒绝写入

### Requirement: 同一聊天同一时刻只能存在一个待处理合作提案

系统 MUST 保证同一个聊天房间在任意时刻只能存在一个待处理试课合作提案。

#### Scenario: 已有待处理提案时再次发起
- **WHEN** 某聊天房间已有状态为 `PENDING` 的合作提案
- **THEN** 系统 MUST 拒绝新的合作提案创建

### Requirement: 合作提案需设置12小时有效期

系统 MUST 为试课合作提案设置 12 小时有效期，并在超时后自动过期。

#### Scenario: 合作提案超时未处理
- **WHEN** 合作提案创建后 12 小时内对方未处理
- **THEN** 系统 MUST 将提案状态更新为 `EXPIRED`
- **AND** 聊天中 MUST 发送提案已过期的系统消息

### Requirement: 合作提案同意后创建试课

系统 MUST 在合作提案被接收方同意后创建试课课程阶段与双方试课日程。

#### Scenario: 同意合作提案
- **WHEN** 接收方同意合作提案
- **THEN** 系统 MUST 创建或更新长期课程
- **AND** 将课程阶段更新为 `TRIAL_SCHEDULED`
- **AND** 在双方日程中写入带“试课”标签的试课事件
- **AND** 聊天中 MUST 发送“试课已安排”的系统消息
