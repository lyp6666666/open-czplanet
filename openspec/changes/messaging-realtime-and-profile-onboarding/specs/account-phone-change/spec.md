## ADDED Requirements

### Requirement: 双端提供修改手机号入口
系统 MUST 在教师端与学生端均提供“修改手机号”的入口，并展示当前已绑定手机号（可脱敏显示）与变更流程说明。

#### Scenario: 教师与学生均可进入修改手机号
- **WHEN** 用户已登录且进入“我的/设置”页面
- **THEN** 页面 MUST 展示“修改手机号”入口，且教师与学生均可访问

### Requirement: 修改手机号必须进行旧号与新号双重校验
系统 MUST 采用两步校验流程：先验证当前绑定手机号可接收验证码（旧号校验），再验证新手机号可接收验证码（新号校验），最后才允许更新绑定关系。

#### Scenario: 先校验旧号再绑定新号
- **WHEN** 用户发起修改手机号，先完成旧号验证码校验，再输入新手机号并完成新号验证码校验
- **THEN** 系统 MUST 将账号手机号更新为新手机号，并返回成功结果

### Requirement: 新手机号必须可用且格式合法
系统 MUST 校验新手机号格式合法；当新手机号已被其他账号绑定时，系统 MUST 拒绝变更并返回明确错误。

#### Scenario: 新手机号被占用时拒绝
- **WHEN** 用户输入的新手机号已被其他用户绑定
- **THEN** 系统 MUST 返回“手机号已被占用”的错误，且不更新当前账号手机号

### Requirement: 修改手机号成功后会话状态必须安全
修改手机号成功后，系统 MUST 采取安全策略保证账号会话安全（例如清理旧手机号维度的 token/会话并要求重新登录，或刷新 token 并要求前端更新本地凭证）。

#### Scenario: 修改成功后要求重新登录
- **WHEN** 用户修改手机号成功
- **THEN** 前端 MUST 引导用户重新登录或刷新登录态，且旧手机号对应的登录态 MUST 不再可用
