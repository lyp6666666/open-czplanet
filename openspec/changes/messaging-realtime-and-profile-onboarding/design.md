## Context

当前前端（`ai-tutor-web`）的消息红点与实时渲染依赖一个全局 SSE 连接（`/chat/stream`，由 `chatRealtime` store 管理）以及聊天页内的额外 SSE 连接（`ChatRoomPage.vue` 自行 fetch 同一 stream）。后端（`videoCall-IM-service`）已提供已读 ACK 接口与“会话已读状态表”的 upsert 逻辑，但实现中对建表与写入异常进行了吞掉处理，导致在数据库未就绪/权限不足/DDL 被禁用时，ACK 可能静默失败，从而出现“本次看过红点消失、刷新又回来”的体验问题。

修改手机号能力后端已有基础接口（校验旧号验证码并更新），但缺少“新手机号验证码校验/占用校验/更接近主流产品的两段式流程”，且前端缺少入口与流程页。

家教首次注册当前仅有“基础信息页”（头像、真实姓名、学历），并通过本地 localStorage 做简单 gating；需求希望拆分为两步（第一步必填：头像+真实姓名；第二步可跳过：简历完善字段），并补齐数据库字段与回显。

## Goals / Non-Goals

**Goals:**

- 消息未读红点在跨页面与刷新后保持一致：以服务端持久化已读状态为准，前端本地状态仅做实时增量
- 消息“主动渲染”：收件方在消息列表/会话页无需刷新即可看到新消息（列表可更新预览与排序，会话页可即时追加）
- 双端提供修改手机号：包含旧号校验与新号校验、错误提示与必要的风控/占用校验，流程尽量贴近主流软件
- 家教首次注册改为两步引导：第一步必填并作为入驻完成的硬门槛；第二步可跳过但支持后续在“简历/我的”补齐
- 涉及数据表结构变更时，提供明确迁移策略与兼容方案（历史数据、默认值、回滚）

**Non-Goals:**

- 不在本次变更中引入 WebSocket/第三方 IM SDK（继续使用现有 SSE + MQ 架构）
- 不在本次变更中对消息加密、敏感词、风控策略做系统性升级（仅做必要的参数校验与错误提示）
- 不在本次变更中重做“个人中心/简历”整体信息架构（只新增本需求涉及字段与流程）

## Decisions

1) **将 `room_read_state` 作为正式迁移管理，而不是运行时建表**

- 选择：在 `sqlDoc/migrations/` 中新增建表脚本，部署前由迁移系统执行；后端移除/停止依赖运行时 `CREATE TABLE IF NOT EXISTS`。
- 理由：运行时 DDL 往往在生产环境被禁用或权限不足；当前实现吞异常会导致 ACK 静默失败，从而出现“刷新又红”的核心问题。
- 备选：继续运行时建表并记录日志。缺点：无法保证成功率与可观测性，且与数据库治理冲突。

2) **ACK 写入失败不再静默吞掉**

- 选择：后端 ACK 接口在写入失败时返回明确错误码；前端 ACK 做幂等重试（现有 `ackPendingByRoom` 队列可复用）。
- 理由：已读状态属于核心一致性路径，静默失败会直接伤害体验；显式失败可快速定位与告警。
- 备选：吞异常但在前端“刷新未读”兜底。缺点：仍会反复闪现红点，且排障困难。

3) **统一使用全局 SSE 连接，避免聊天页重复连接**

- 选择：将聊天页内独立的 `/chat/stream` 连接收敛到 `chatRealtime` store；聊天页通过监听 `chatRealtime.lastEvent` 追加消息并 ACK。
- 理由：避免同一用户多连接带来的重复消息、双重 ACK、资源浪费与难以排查的时序问题；全局连接更易做重连与可观测。
- 备选：保留聊天页独立连接。缺点：两套解析/重连逻辑长期漂移，问题复现困难。

4) **消息列表实时更新策略：事件驱动 + 轻量回源**

- 选择：收到 `message` 事件时，前端先用事件携带的 `msgId/sendTime/body/fromUid/toUid` 更新列表“预览/排序/未读”；必要时（缺字段或需要强一致）再对单房间做一次回源拉取（例如重新请求 `/chat/room/page` 的首屏或提供轻量 `GET /chat/room/{roomId}` 端点）。
- 理由：列表体验需要“及时可见”，但不必每条都全量刷新；轻量回源可兜底一致性与字段缺失。
- 备选：每条消息都全量刷新房间列表。缺点：开销大、抖动明显。

5) **修改手机号采用“两段验证码”流程，并保持接口兼容**

- 选择：新增“发送新手机号验证码”与“提交变更（校验旧号 + 新号验证码）”的接口；保留现有 `updateUserPhone` 作为兼容入口（或内部转调新逻辑）。
- 理由：主流体验通常需要确认当前账号身份与新号码可达性；同时减少误操作与盗号风险。
- 备选：只校验旧号验证码后直接改为新号（现状）。缺点：无法验证新号可达，且更易被输入错误号码导致账号不可用。

6) **家教两步入驻：第一步硬门槛，第二步软引导**

- 选择：第一步提交后写入头像+真实姓名，并在服务端记录“基础完成”；第二步写入简历字段并标记“简历完成”，同时提供“跳过”仅结束引导不阻塞使用。
- 理由：确保最关键的可信信息先完善；更长的简历信息允许用户分阶段补齐，降低流失。
- 备选：继续用 localStorage gating。缺点：换设备/清缓存会反复引导，且与服务端状态不一致。

## Risks / Trade-offs

- [生产环境迁移/权限问题导致已读表不可用] → 在迁移上线前做预检查；ACK 接口显式报错并纳入监控，避免“静默错”
- [统一 SSE 连接后，聊天页消息追加的时序与去重] → 以 `msgId` 做去重；ACK 使用 `max(lastReadMsgId)` 幂等上报
- [列表实时更新字段不足（例如缺少对端姓名/头像）] → 复用现有用户卡片缓存；必要时对缺失用户信息做懒加载
- [修改手机号引入新流程影响已有客户端] → 保留旧接口兼容；新端优先走新接口；灰度期间双写/兜底
- [新增简历字段影响历史数据展示与筛选] → 新字段默认 NULL；展示层做空值兜底；筛选/排序功能后续单独迭代

## Migration Plan

1. 数据库
   - 新增 `room_read_state` 正式建表迁移（若线上已存在表则脚本需幂等）
   - 教师档案表新增字段（所在城市、最高学历学校、教学方式等）并提供默认值/允许 NULL
2. 后端
   - 调整 ACK 逻辑：移除运行时建表依赖；写入失败返回错误
   - 增加修改手机号新接口，并将旧接口兼容映射到新流程（或保留旧流程但增加新号校验）
   - 扩展用户信息更新接口以支持新增的教师简历字段
3. 前端
   - 收敛 SSE：聊天页移除重复连接，统一使用全局 store 的事件
   - 消息列表/红点：用事件驱动更新列表预览与未读，同时定期/关键路径回源校准
   - 新增修改手机号页面与入口（教师/学生均可）
   - 家教两步引导页：替换现有基础页，完成后写入服务端状态并更新路由守卫
4. 回滚
   - 前端可直接回滚到旧版本（接口保持兼容）
   - 后端新接口可保留不影响旧端；数据库新增字段不影响旧逻辑（允许 NULL）

## Open Questions

- 已读 ACK 的语义：以“最后读到的消息 id”为准是否足够，是否需要补充“进入会话即已读到最新”策略与防误读（例如消息加载尚未完成时）
- 修改手机号是否需要强制重新登录（现状会清理旧手机号 token），以及 token 维度是按手机号还是按 uid 存储（需结合现有登录实现确认）
- 擅长学科多选在后端的存储格式：继续复用 `subject` 字符串（逗号分隔）还是升级为 JSON 字段；本次倾向于先用字符串兼容落地
