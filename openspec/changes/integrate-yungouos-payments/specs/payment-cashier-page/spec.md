## ADDED Requirements

### Requirement: 系统提供收银台新开页用于扫码支付
系统 MUST 提供独立的收银台页面，并支持从业务页面通过新开浏览器标签页进入，用于展示订单信息与扫码支付二维码。

#### Scenario: 业务页新开标签页进入收银台
- **WHEN** 用户在业务页面点击“去支付”
- **THEN** 系统 MUST 以新开浏览器标签页的方式打开收银台页面，并携带用于定位业务订单的参数

### Requirement: 收银台支持微信/支付宝两种扫码方式切换
收银台页面 MUST 提供微信与支付宝两种支付方式切换，并在切换后展示对应的二维码与指引信息。

#### Scenario: 切换到微信支付展示微信二维码
- **WHEN** 用户在收银台选择“微信支付”
- **THEN** 系统 MUST 请求后端统一下单并展示微信支付二维码，同时展示扫码指引

#### Scenario: 切换到支付宝支付展示支付宝二维码
- **WHEN** 用户在收银台选择“支付宝支付”
- **THEN** 系统 MUST 请求后端统一下单并展示支付宝支付二维码，同时展示扫码指引

### Requirement: 收银台必须轮询查询支付状态并实时反馈
收银台页面 MUST 通过后端“支付单查询”接口轮询订单状态，并在支付成功、失败或过期时给出明确反馈。

#### Scenario: 支付成功后展示成功状态并引导返回
- **WHEN** 轮询查询到支付单状态为成功
- **THEN** 收银台 MUST 展示支付成功结果，并通知打开收银台的业务页面刷新业务状态或引导用户返回

#### Scenario: 订单过期后停止轮询并提示刷新二维码
- **WHEN** 支付要素到达过期时间且订单仍未支付成功
- **THEN** 收银台 MUST 停止轮询并提示用户刷新二维码或重新发起支付

### Requirement: 收银台展示信息与安全约束
收银台 MUST 展示订单号、金额与支付说明等必要信息，并避免在前端暴露任何支付密钥或可伪造支付成功的敏感字段。

#### Scenario: 页面不展示密钥与签名敏感信息
- **WHEN** 收银台渲染支付信息
- **THEN** 页面 MUST 不展示任何支付密钥、签名串、验签材料等敏感信息
