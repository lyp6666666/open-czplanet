## Why

当前项目已存在支付模块与回调入口，但缺少“统一下单/出码/查单”等闭环关键能力，前端仍以“收款码+提交凭证+人工确认”为主，导致线上支付不可用、不可追溯、难以对账与自动解锁业务。需要引入 YunGouOS 第三方支付体系，按推荐流程实现微信/支付宝支付闭环，并补齐管理端支付记录查询与测试保障。

## What Changes

- 后端引入 `com.yungouos.pay:yungouos-pay-sdk:2.0.37`，以 YunGouOS 为支付聚合提供方，实现微信/支付宝扫码支付的统一下单与支付要素生成（二维码/链接）。
- 新增支付域 API：统一下单（prepay/出码）、支付单查询（供前端轮询）、YunGouOS 异步通知回调（验签、金额/商户信息校验、幂等落库、状态迁移、触发业务事件）。
- 完善支付闭环关键数据落库：记录支付单生命周期关键字段（渠道、金额、业务关联、第三方单号/交易号、回调接收信息、成功时间、过期时间等），并给出数据库设计说明。
- 打通“支付成功事件”到业务域：支付成功后发布事件，业务服务消费事件后将业务订单置为已支付并解锁后续流程（严格幂等）。
- 前端新增“收银台新开页”并优化交互：在新浏览器标签页展示订单信息、微信/支付宝两种扫码方式切换、二维码展示、支付状态轮询与支付成功引导返回（对齐主流网站收银台体验）。
- 管理员端新增“付款记录查看”：支持按订单号/业务类型/业务ID/用户/渠道/状态/时间范围检索支付订单，并查看回调与状态变更关键信息（用于对账与排障）。
- 全链路测试与质量：补齐单元测试（签名/验签、状态机幂等、回调处理）、集成测试（Mock YunGouOS 回包与回调）、流程闭环测试（下单→出码→回调→状态更新→事件→业务解锁），并要求关键实现具备完善中文注释与日志（脱敏）。

## Capabilities

### New Capabilities
- `yungouos-scan-pay`: 基于 YunGouOS SDK 的微信/支付宝扫码支付统一下单与出码能力。
- `yungouos-notify-closed-loop`: YunGouOS 异步通知回调验签、幂等更新、关键字段校验、状态迁移与事件发布的闭环能力。
- `payment-cashier-page`: 新开页收银台（微信/支付宝切换、二维码展示、轮询查单、成功/失败/过期处理）。
- `admin-payment-records`: 管理员端付款记录查询与详情查看能力（含必要审计信息与排障字段）。
- `payment-e2e-tests`: 支付闭环的测试能力（单测+集成测+流程测试，覆盖主要分支与幂等场景）。

### Modified Capabilities
- `brokerage-payment-qrcode`: 由“静态收款码+提交凭证+人工确认”升级为“服务端统一下单出码+回调闭环+前端轮询确认”，保留必要的异常兜底与兼容策略。
- `payment-order-management`: 支付订单管理能力扩展为支持 YunGouOS 渠道字段、第三方单号/交易号、回调接收信息与更严格的状态机/幂等要求。

## Impact

- Affected backend services:
  - `payment-service`：新增统一下单/查单/回调闭环能力，扩展支付订单字段与状态迁移逻辑，补齐事件发布与幂等控制。
  - `videoCall-IM-service`（或业务域服务）：消费支付成功事件并将 `brokerage_order` 等业务订单置为已支付与解锁（幂等）。
- Affected frontend:
  - `ai-tutor-web`：新增收银台新开页、替换/升级原中介费支付页入口与交互，新增轮询与支付完成回跳处理。
  - `ai-tutor-admin`：新增付款记录查看页面与后端 API 对接。
- Data & migrations:
  - `payment_order` 表可能需要新增字段（第三方单号/交易号、支付要素、回调信息、过期时间/成功时间等）与索引（按业务关联/订单号/状态/时间查询）。
- Ops & security:
  - 新增回调公网可达地址与验签密钥配置；回调接口需限流、日志脱敏、幂等与告警指标（回调验签失败、长时间未支付等）。
