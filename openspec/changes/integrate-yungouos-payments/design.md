## Context

### 背景与现状

- 当前代码库已存在支付域服务 `payment-service`，包含支付订单表 `payment_order` 及支付宝/微信回调入口，但缺少“统一下单/出码/查单”能力，无法形成可用的支付闭环。
- 业务侧（IM/撮合）存在中介费订单 `brokerage_order`，前端支付页当前以“收款码展示 + 上传凭证 + 管理端人工确认”为主，无法做到自动对账与自动解锁业务。
- 代码库已经具备“支付成功事件 → 业务解锁”的消费逻辑，但上游没有创建并正确关联 `payment_order.contextType/contextId`，导致事件链路无法生效。

### 约束与目标流程

- 需要引入 YunGouOS（`yungouos-pay-sdk:2.0.37`）作为聚合支付信息流提供方，实现微信/支付宝扫码支付。
- 必须严格按推荐流程实现闭环：前端发起购买 → 服务端统一下单拿到二维码 → 前端展示并轮询 → 用户支付成功 → YunGouOS 异步通知服务端 → 服务端验签/校验/幂等入库 → 触发业务逻辑（事件） → 前端最终展示成功。
- 代码质量要求：关键逻辑具备完善中文注释；单元测试覆盖核心分支；提供端到端的闭环测试策略；关键业务与风控信息落库；数据库设计合理且给出说明；管理员端可查询付款记录。

## Goals / Non-Goals

**Goals:**
- 在 `payment-service` 内提供统一下单（prepay/出码）、支付单查询、YunGouOS 回调闭环处理三大能力，形成可用支付闭环。
- 支付订单与业务订单强关联：支持以 `contextType/contextId` 关联 `brokerage_order`，支付成功后发布事件，业务服务消费事件完成自动解锁。
- 前端新增“收银台新开页”：支持微信/支付宝两种扫码方式切换、二维码展示、倒计时、轮询查单、成功引导返回（对齐主流网站体验）。
- 管理员端新增付款记录查看：支持检索、分页、查看详情字段，用于对账与排障。
- 测试覆盖：核心逻辑单测 + 回调幂等/状态机单测 + Mock YunGouOS 的集成测试 + 闭环流程测试。

**Non-Goals:**
- 本变更不覆盖退款、撤销、分账、对账单下载等高级支付能力（后续可扩展）。
- 不实现 App 支付/小程序支付/公众号支付等非扫码场景（如未来需要将以同一支付域框架扩展）。
- 不在本变更中重构既有鉴权体系，仅在新增接口遵循当前项目的鉴权/网关方式并补齐必要的安全校验。

## Decisions

### 1) 支付聚合：采用 YunGouOS SDK + 服务端统一下单

**决策：**
- 支付域统一对接 YunGouOS SDK，由后端生成支付要素（二维码链接/支付链接），前端仅负责展示与轮询，不直接拼接支付签名参数。

**原因：**
- 统一收敛签名与参数拼装，降低安全风险与接入成本。
- 满足推荐流程中“商户 Server 构造参数请求支付二维码 → 前端展示二维码”的闭环模式。

**备选方案：**
- 直接对接微信/支付宝官方 API（已存在回调入口）：接入复杂度高、维护成本高、需要分别处理多套签名与证书体系，不符合本次“第三方支付体系”要求。

### 2) 订单模型：以 `payment_order` 为支付域单一事实来源（SSOT）

**决策：**
- `payment_order` 作为支付域的权威记录，支付成功以“回调验签通过并落库成功”为准。
- 业务解锁只由“支付成功事件”驱动，业务侧不得以前端状态作为最终依据。

**原因：**
- 防止前端伪造支付成功。
- 便于幂等处理与审计追踪。

**备选方案：**
- 业务服务直接处理回调并更新业务订单：会导致支付逻辑散落，难以复用，且与现有 `payment-service` 的分层不一致。

### 3) 回调闭环：强验签 + 关键字段二次校验 + 幂等状态机

**决策：**
- YunGouOS 回调必须完成：验签 → 查单 → 金额/渠道/商户号校验 → 幂等更新 `payment_order` → 发布事件。
- 对同一支付单的重复回调必须幂等，确保最多一次触发业务事件。

**原因：**
- 回调天然存在重试与重复投递；必须保障最终一致性与“至多一次业务副作用”。

**备选方案：**
- 仅验签不校验金额/商户信息：存在串单/篡改风险，不符合上线要求。

### 4) 前端体验：采用“新开页收银台 + 轮询查单”方案

**决策：**
- 业务页点击“去支付”后 `window.open()` 打开新标签页收银台。
- 收银台页：默认展示微信扫码，可切换支付宝扫码，内部轮询支付状态；成功后通过 `postMessage` 通知 opener 刷新业务状态并提示用户返回。

**原因：**
- 对齐主流网站的支付体验（避免在原页面路由栈中阻塞或被拦截）。
- 扫码支付需要稳定展示二维码与支付状态反馈，新页面隔离更清晰。

**备选方案：**
- 原页面内弹窗/抽屉：对移动端与浏览器拦截策略更敏感，且路由与状态回收复杂。

### 5) 管理端付款记录：以支付域数据为主，提供检索与详情

**决策：**
- 管理端查询直接访问 `payment-service` 的管理 API（或由 admin 服务透传/聚合，取决于当前网关与服务治理方式），支持分页与多条件筛选。
- 明确“对账/排障字段”与“敏感字段”边界：不展示签名密钥、不落库密钥、不输出回调原文中的敏感字段（必要时仅落库脱敏后的摘要）。

**原因：**
- 支付域数据最完整，适合统一审计与排障。

## Risks / Trade-offs

- **[回调安全风险]** 回调接口公网暴露 → 需验签、限流、日志脱敏、必要时加 IP 白名单/网关鉴权（若 YunGouOS 支持固定出口）。
- **[金额单位风险]** YunGouOS API 参数通常为“元”字符串，而库内常为“分”整数 → 统一在支付域做金额换算与双向校验，单测覆盖边界。
- **[重复下单风险]** 用户反复点击“去支付”导致多笔待支付单 → 采用“同业务订单+同渠道幂等复用，过期后再新建”的策略，避免支付侧混乱。
- **[事件重复消费]** MQ 重投递导致业务重复解锁 → 业务服务消费端按 `orderNo` 或 `contextId` 做幂等，且支付域只在 `PENDING→SUCCESS` 时发布一次事件。
- **[兼容旧流程]** 现有“上传凭证+人工确认”可能仍被依赖 → 设计为可配置的兜底开关：仅在回调不可用或支付渠道不可用时启用，默认收敛到新闭环。

## Migration Plan

### 数据库变更

- 为 `payment_order` 增加必要字段与索引（详见 specs 中的数据字典与迁移脚本），目标是支撑：
  - YunGouOS 第三方单号/交易号记录
  - 支付要素（二维码地址/支付链接）记录与过期时间
  - 回调接收信息（接收次数、最后接收时间、验签结果、摘要）用于排障
  - 便于管理端按状态/时间范围/业务关联/用户筛选的索引

### 发布步骤（建议）

1. 先上线 DB migration（新增字段/索引），保持向后兼容（旧代码不使用新增字段也可运行）。
2. 上线 `payment-service` 新接口：prepay/查单/回调（回调地址先配置但不切流）。
3. 灰度切换前端入口到新收银台页（按用户/环境开关）。
4. 确认回调可达与验签通过后，逐步切换默认支付流程到新闭环。
5. 管理端上线付款记录查询，便于观察真实数据与排障。

### 回滚策略

- 若 YunGouOS 回调或下单异常，可通过配置开关将前端入口回退到旧的“凭证提交/人工确认”流程，保证业务可用性。
- 数据库新增字段不影响旧逻辑，无需回滚；仅需停用新接口路由或关闭灰度开关。

## Open Questions

- YunGouOS 回调验签所需的密钥/方式在本项目将以何种配置形态提供：配置中心、环境变量还是数据库？（倾向：配置中心 + 分环境隔离 + 不落库密钥）
- 管理端鉴权与审计：当前 admin 的鉴权方式与操作日志沉淀策略是什么？（需要对接既有 admin-auth 能力，避免继续使用静态 Token）
- 扫码支付在 PC 与移动端的兼容策略：是否需要同一收银台页面根据 UA 自动切换到 H5 拉起支付模式？（本次默认 PC 扫码；如需移动端增强可作为扩展）
