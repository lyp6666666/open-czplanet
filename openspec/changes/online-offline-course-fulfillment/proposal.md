## Why

当前平台的核心闭环仍以“信息费支付后解锁沟通”为主，尚未区分线下撮合与线上履约两种完全不同的业务模型，导致产品承诺、用户预期、课程管理、支付结算与聊天权限都混在一起。随着平台要承接线上实时一对一授课，现有流程已经不足以支撑长期课程、逐节支付、试课判断、课程表滚动管理和教师结算。

同时，授课形式一旦被学生选定，就会直接影响后续订单结构、服务承诺、退款规则、支付路径、课表能力和平台责任边界。如果允许中途切换授课形式，会引发价格、协议、聊天权限、课程历史和支付对账的混乱。因此本次需要把“授课形式分流”提升为整个业务链路的第一层分水岭，并明确授课形式创建后不可切换。

## What Changes

- 新增“授课形式分流”能力，在“发布需求”和“主动联系教师”两个入口强制选择 `线上` 或 `线下`
- 明确授课形式是强业务字段：需求、沟通、课程、订单创建后 MUST 继承首选授课形式，且 MUST NOT 支持切换
- 为线下模式保留现有“一次性信息费 + 平台不参与后续履约”的模式
- 为线上模式新增长期课程与短期课节体系，支持试课、滚动课表、加课、删课、调课、暂停与结束课程
- 为线上模式新增“课后支付与教师结算”闭环，学生逐节支付，平台从教师侧抽取 10% 服务费，学生端不感知服务费拆分
- 为线上模式新增“试课退课与聊天权限治理”闭环，试课后学生必须选择继续或退课；退课后课程结束、后续课节取消、聊天关闭
- 输出面向研发的执行计划，按以下四个模块拆解：
  - 授课形式分流
  - 线上课程/课节系统
  - 课后支付结算
  - 试课退课与聊天权限
- 将 Git 提交策略、中文注释要求和端到端全流程测试纳入交付要求

## Capabilities

### New Capabilities

- `teaching-mode-routing`: 授课形式分流与不可切换约束
- `online-course-and-lesson-system`: 线上长期课程/短期课节、课表与排课管理
- `post-class-payment-settlement`: 逐节课后支付、平台服务费结算与阻塞规则
- `trial-refund-and-chat-governance`: 试课判断、退课闭环与聊天权限治理

### Modified Capabilities

- `teacher-demand-chat-entry`: 在线上沟通通过后增加“创建课程”入口与课程卡片
- `payment-processing`: 增加课节支付、支付成功后驱动课程状态推进
- `payment-order-management`: 增加课程课节订单、教师分账/结算记录

## Impact

- 前端：
  - `ai-tutor-web` 需要改造发布需求页、教师详情页、聊天页、我的课程页、课程详情页、课节详情页、支付页与试课退课页
- 后端：
  - `tutor-appointment-service` 需要承接需求授课形式字段、不可切换校验与学生/教师资料透出
  - `videoCall-IM-service` 需要承接线上沟通、课程创建、聊天权限关闭与课程入口联动
  - `payment-service` 需要承接线上课节支付、教师结算与支付阻塞规则
  - `live-class-service` 需要与课程课节联动，承接线上课堂准入与课后完成事件
- 数据库：
  - 新增或扩展需求、课程、课节、支付订单、结算单、试课结果、聊天权限状态等表结构
- 测试：
  - 补齐模块级单测/集成测试
  - 提供线上完整业务闭环的端到端测试：选授课形式 → 沟通 → 建课 → 试课 → 课后支付 → 下一节解锁/退课关闭
