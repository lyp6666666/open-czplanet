# 一、授课形式分流

- [ ] 发布需求时必须选择授课形式，且文案明确区分线上/线下
- [ ] 学生主动联系教师时必须弹窗选择授课形式
- [ ] 已创建的需求、沟通、课程不可切换授课形式
- [ ] 前端不存在“切换线上/线下”的入口
- [ ] 后端更新接口对授课形式重写返回明确错误

# 二、线下模式

- [ ] 线下仍走一次性信息费流程
- [ ] 线下聊天解锁后，不进入线上课程/课节系统
- [ ] 线下页面明确提示平台不参与后续质量保证和课后支付

# 三、线上课程/课节系统

- [ ] 沟通通过后可发起创建长期课程
- [ ] 课程创建后自动生成第一节试课
- [ ] 我的课程页展示的是长期课程而非单节课
- [ ] 进入课程详情后可查看课节列表
- [ ] 支持加课、删课、调课、暂停课程
- [ ] 修改单节课与修改长期课表的影响范围清晰可控

# 四、课后支付结算

- [ ] 每节线上课完成后生成待支付订单
- [ ] 学生端仅展示本节应付课时费，不展示 10% 服务费拆分
- [ ] 教师端可看到平台服务费、预计到账金额和可提现余额
- [ ] 学生未支付上一节课时，无法预约下一节
- [ ] 学生未支付上一节课时，无法进入下一节线上课堂
- [ ] 学生支付成功后，下一节恢复可预约/可进入

# 五、试课退课与聊天权限

- [ ] 第一节课默认为试课
- [ ] 试课完成并支付后，学生必须选择“合适”或“不合适”
- [ ] 选择“合适”后，课程进入正常进行中状态
- [ ] 选择“不合适”后，课程结束且未开始课节全部取消
- [ ] 选择“不合适”后，聊天关闭或转只读，无法继续发送消息
- [ ] 退课原因可被记录，便于后续运营分析

# 六、端到端全流程测试

- [ ] 线上继续链路 E2E 跑通
- [ ] 线上退课链路 E2E 跑通
- [ ] 未支付阻塞链路 E2E 跑通
- [ ] 前端关键页面自动化验证通过
- [ ] 后端单测、集成测试和跨服务 E2E 全部通过

# 七、交付规范

- [ ] 关键业务代码补有中文注释，解释状态机、阻塞规则和不可切换约束
- [ ] 代码按四个模块分阶段 Git 提交
- [ ] 提交信息使用中文
- [ ] 提交与测试记录可追溯
