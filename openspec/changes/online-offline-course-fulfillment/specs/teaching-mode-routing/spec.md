## ADDED Requirements

### Requirement: 用户必须先选择授课形式
系统 MUST 在学生发布需求和学生主动联系教师两个关键入口要求用户先明确选择授课形式，取值仅为 `ONLINE` 或 `OFFLINE`。

#### Scenario: 发布需求时选择授课形式
- **WHEN** 学生提交新的授课需求
- **THEN** 系统 MUST 要求学生在提交前选择 `ONLINE` 或 `OFFLINE`

#### Scenario: 主动联系教师时选择授课形式
- **WHEN** 学生在教师详情页点击联系教师或发起沟通
- **THEN** 系统 MUST 弹出授课形式选择层，并在提交沟通请求时携带该选择结果

### Requirement: 授课形式创建后不可切换
系统 MUST 将授课形式视为强业务约束字段。需求、沟通关系、长期课程一旦创建成功，授课形式 MUST NOT 被修改。

#### Scenario: 前端尝试修改授课形式
- **WHEN** 用户进入需求详情、聊天页、课程页等后续页面
- **THEN** 系统 MUST NOT 展示切换授课形式的交互入口

#### Scenario: 后端收到授课形式修改请求
- **WHEN** 客户端对已存在的需求、沟通关系或课程提交不同于原值的 `teachingMode`
- **THEN** 系统 MUST 拒绝该请求，并返回明确的业务错误码和提示信息

### Requirement: 线下模式仅承接撮合信息服务
系统 MUST 将线下模式限定为信息撮合服务，并 MUST NOT 将线下沟通自动纳入线上课程履约体系。

#### Scenario: 线下沟通成功
- **WHEN** 学生以 `OFFLINE` 形式发起沟通并完成信息费支付
- **THEN** 系统 MUST 仅解锁沟通，不创建长期课程、课节订单或线上支付任务

### Requirement: 线上模式必须进入课程履约流程
系统 MUST 将线上模式的沟通结果引导至长期课程创建，并为后续课程、课节、支付和试课状态提供承接关系。

#### Scenario: 线上沟通通过
- **WHEN** 学生以 `ONLINE` 形式发起沟通且教师通过
- **THEN** 系统 MUST 在聊天页或相关入口提供“创建课程”能力，并将后续履约行为绑定到长期课程
