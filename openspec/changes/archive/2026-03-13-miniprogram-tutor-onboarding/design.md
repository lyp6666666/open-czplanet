# 设计：小程序教师入驻流程

## 背景 (Context)
Web 端已经存在完整的教师入驻流程，包含多个步骤的表单提交。小程序端目前仅支持学生端功能和简单的角色切换。为了支持教师在移动端入驻，需要移植 Web 端的表单逻辑，适配移动端交互。

## 目标 / 非目标 (Goals / Non-Goals)

**目标：**
- 实现分步表单（Stepper），引导用户完成入驻。
- 支持头像上传、文本输入、选择器（科目、年级）等表单控件。
- 处理入驻状态（未提交 -> 审核中 -> 已通过/已拒绝）。
- 复用后端 `/tutor/apply` 接口。

**非目标：**
- 实现复杂的证件 OCR 识别（移动端仅支持图片上传）。
- 修改后端入驻逻辑。

## 决策 (Decisions)

### 1. 表单结构
- **决策**：采用分步表单（Step 1: 基本信息, Step 2: 教学偏好, Step 3: 教育背景）。
- **理由**：移动端屏幕较小，长表单体验不佳，分步能降低用户心理负担。
- **组件**：使用 `uView Plus` 的 `u-steps` 和 `u-form`。

### 2. 状态管理
- **决策**：在 `userStore` 中增加 `tutorStatus` 状态。
- **逻辑**：
  - 登录时获取用户扩展信息，检查 `teacherProfile` 是否存在及状态。
  - 点击“切换为教师”时，如果状态为 `NONE`，跳转到入驻页。
  - 如果状态为 `PENDING`，显示审核中页面。
  - 如果状态为 `REJECTED`，显示拒绝原因并允许重新提交。
  - 如果状态为 `APPROVED`，进入教师工作台。

### 3. 图片上传
- **决策**：使用 `uni.uploadFile` 上传头像和证书，后端复用通用的文件上传接口。

## 风险 / 权衡 (Risks / Trade-offs)
- **[风险]**：表单项较多，移动端输入效率低。
  - *缓解措施*：尽量使用选择器（Picker）代替输入框，提供默认值。

## 迁移计划
1.  封装通用上传组件。
2.  开发入驻分步页面。
3.  修改 `userStore` 状态判断逻辑。
4.  联调测试。
