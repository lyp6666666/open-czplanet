# 设计：微信小程序 Mock 集成与业务完善

## 背景 (Context)
当前微信小程序端的开发受限于必须使用真实 AppID 和商户号，导致开发人员无法在本地环境中完整跑通登录和支付流程。此外，预约功能目前仅实现了简单的“直接订单”，缺乏与具体预约时间的关联，业务闭环不完整。

## 目标 / 非目标 (Goals / Non-Goals)

**目标：**
- 实现一套 Mock 机制，允许在无真实微信账号的情况下完成登录和支付流程。
- 完善导师预约流程，支持选择时间并创建关联预约的订单。
- 确保 Mock 逻辑与真实业务逻辑解耦，通过配置开关控制。

**非目标：**
- 实现复杂的预约调度算法（如冲突检测），仅关注流程跑通。
- 实现真实的微信支付回调验证（Mock 模式下直接通过）。

## 决策 (Decisions)

### 1. Mock 登录策略
- **决策**：在 `WechatAuthService` 中通过 `appid` 判断是否启用 Mock。
- **理由**：
  - 如果配置的 `appid` 为特定值（如 `wx_mock_appid`），则认为处于 Mock 模式。
  - 这种方式不需要修改代码结构，只需调整配置文件即可切换环境。
- **替代方案**：使用单独的 `MockWechatAuthService` 实现类。虽然更符合开闭原则，但在简单场景下增加了类的数量，且切换需要依赖注入配置。

### 2. Mock 支付策略
- **决策**：修改 `PaymentAppService` 和前端支付逻辑。
- **后端**：在 `prepay` 方法中，如果检测到 Mock 模式，生成一个带有特殊标记（如 `mock: true`）的 `payParams`。
- **前端**：检查 `payParams`，如果包含 Mock 标记，则不调用 `uni.requestPayment`，而是直接模拟成功回调。
- **理由**：这种方式对前端代码侵入最小，且能覆盖除最终微信调用外的所有业务逻辑。

### 3. 预约与订单关联
- **决策**：
  - 前端先调用 `TutorAppointmentService` 创建预约，获取 `appointmentId`。
  - 前端调用 `BrokerageOrderService` 创建订单时，将 `appointmentId` 作为参数传递（或存储在 `extra` 字段中）。
  - 后端在创建 `BrokerageOrder` 时记录关联信息。
- **理由**：保持服务职责单一，`TutorAppointment` 负责调度，`BrokerageOrder` 负责支付。

## 风险 / 权衡 (Risks / Trade-offs)
- **[风险]**：Mock 代码意外进入生产环境。
  - *缓解措施*：通过配置严格控制 Mock 开关，确保生产环境配置为真实 AppID。
- **[风险]**：Mock 流程与真实流程差异过大，导致部分 Bug 未被发现。
  - *缓解措施*：尽量保持 Mock 逻辑仅替换最底层的微信 API 调用，上层业务逻辑保持一致。

## 迁移计划
1. 修改后端配置，启用 Mock 模式。
2. 更新后端服务代码，实现 Mock 逻辑。
3. 更新前端代码，适配 Mock 支付和完善预约流程。
4. 自测验证。
