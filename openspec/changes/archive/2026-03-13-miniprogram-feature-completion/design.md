# 设计：小程序功能补全与双端架构

## 背景 (Context)
当前小程序仅实现了家长端（Student）的基础找导师功能。为了对齐 Web 端体验，需要引入教师端（Tutor）功能，支持需求发布与管理，以及双端通用的聊天功能。

## 目标 / 非目标 (Goals / Non-Goals)

**目标：**
- 实现基于用户角色的首页动态展示（家长看导师，教师看需求）。
- 实现家长发布家教需求和管理我的需求。
- 实现教师查看需求广场。
- 实现基础的 IM 聊天功能（列表与详情）。

**非目标：**
- 实现复杂的教师入驻流程（复杂的表单和资质上传建议引流至 Web 端或后续迭代）。
- 实现即时通讯的高级功能（如语音、视频通话，目前仅限文本）。

## 决策 (Decisions)

### 1. 双端架构
- **决策**：在 `pages/home/index` 中根据 `userStore.userInfo.userType` 动态渲染不同的组件。
  - `ParentHome`: 展示导师列表（现有逻辑）。
  - `TutorHome`: 展示需求广场（新逻辑）。
- **理由**：保持单一入口，用户无感切换。
- **数据流**：`userStore` 增加 `currentRole` 状态，登录时初始化。

### 2. 聊天实现
- **决策**：采用**短轮询 (Short Polling)** 机制获取新消息。
  - 在聊天列表页和聊天室内，每隔 5-10 秒调用一次 `listMessages` 或 `listRooms` 接口。
- **理由**：
  - 后端目前使用 SSE (`/chat/stream`) 推送消息，小程序不支持标准的 EventSource。
  - 虽然可以使用 `uni.request` 的 `enableChunked` 模拟，但为了快速稳定上线，轮询是成本最低且最可靠的方案。
  - 用户量较小时，轮询对服务器压力可控。

### 3. 需求发布
- **决策**：移植 Web 端的表单逻辑，使用 `uView Plus` 的 `u-form` 组件重构。
- **简化**：对于复杂的级联选择（如科目、地区），如果组件支持度不够，可简化为单层选择或输入框。

## 风险 / 权衡 (Risks / Trade-offs)
- **[风险]**：轮询导致消息延迟或耗电。
  - *缓解措施*：仅在页面显示时轮询 (`onShow` 开始, `onHide` 停止)。设置合理的轮询间隔（如 5s）。
- **[风险]**：教师入驻流程复杂，小程序体验不佳。
  - *缓解措施*：小程序端仅提供基础信息查看，引导用户去 Web 端完成复杂的入驻流程。

## 迁移计划
1.  **基础架构**：修改 `userStore` 和 `request.ts`，支持角色判断。
2.  **页面开发**：
    -   聊天模块（列表+详情）。
    -   需求模块（发布+列表）。
    -   我的模块（我的需求+申请）。
3.  **首页重构**：集成双端逻辑。
4.  **联调测试**。
