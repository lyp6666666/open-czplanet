# 设计：微信小程序架构

## 背景（Context）
AI 导师平台目前作为一个 Vue 3 Web 应用程序运行。为了进入微信生态系统，我们正在添加一个小程序前端。这需要一个新的前端代码库和后端更新，以支持微信特定的认证和支付。

## 目标 / 非目标（Goals / Non-Goals）
**目标：**
- 推出功能齐全的微信小程序，包含核心功能：浏览导师、预约课程、聊天、支付。
- 复用现有的后端服务（`tutor-appointment-service`，`payment-service`），尽量减少更改。
- 确保在微信上的高性能和原生体验。

**非目标：**
- 将*整个*管理后台移植到小程序（重点是学生/导师客户端）。
- 在此阶段支持其他小程序平台（支付宝、抖音），尽管所选框架允许以后扩展。

## 决策（Decisions）

### 1. 前端框架：Uni-app (Vue 3)
- **决策**：使用 Vue 3 和 TypeScript 的 Uni-app。
- **理由**：
  - **一致性**：现有的 Web 应用程序是 Vue 3 + TS。使用 Uni-app 允许共享逻辑/类型，并减少开发人员的上下文切换。
  - **效率**：“一次编写，到处运行”的能力允许未来扩展到其他平台。
  - **生态系统**：对微信功能的丰富插件支持（uView UI 等）。

### 2. 认证流程
- **决策**：实现原生微信登录（`wx.login`），用 `code` 换取 JWT。
- **流程**：
  1. 客户端调用 `uni.login()` -> 获取 `code`。
  2. 客户端将 `code` 发送到 `/api/auth/wechat/login`。
  3. 后端与微信 API 交换 `code` 获取 `openid` 和 `session_key`。
  4. 后端通过 `openid` 查找/创建用户。
  5. 后端返回 JWT 令牌（与 Web 应用程序格式相同）。
- **替代方案考虑**：WebView 登录。*拒绝*，因为它提供较差的用户体验，并且限制了对原生功能（如支付）的访问。

### 3. 支付集成
- **决策**：复用 `payment-service`，采用新的 `trade_type=JSAPI` 策略。
- **详情**：
  - 小程序需要特定的 `appId`（不同于 App/Web `appId`）。
  - 后端必须使用小程序的私钥对支付参数进行签名。
  - 前端使用带有签名负载的 `uni.requestPayment()`。

### 4. UI 组件库
- **决策**：使用 `uView Plus`（Vue 3 版本）。
- **理由**：专为 Uni-app/小程序设计的综合组件套件，符合“移动优先”的设计需求。

## 风险 / 权衡（Risks / Trade-offs）
- **[风险]**：应用商店审核。教育/导师类的小程序可能需要特定的资质。
  - *缓解措施*：确保选择“教育”类目并准备好所需文件。
- **[风险]**：包体积。Uni-app 可能会产生较大的包。
  - *缓解措施*：在构建设置中启用“最小化”，对不太关键的功能（例如，如果复杂的特定导师详情页面）使用分包。

## 迁移计划
1. **阶段 1**：设置 Uni-app 项目结构和基本导航（Tabbar）。
2. **阶段 2**：实现登录和用户个人资料（只读）。
3. **阶段 3**：导师列表和详情页面。
4. **阶段 4**：预约和支付流程。
5. **阶段 5**：聊天集成（WebSocket）。
6. **发布**：提交审核。
