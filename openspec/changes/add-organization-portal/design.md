## Context

现状：
- Web 端前台（学生/教师）为同一个 SPA（`ai-tutor-web`），通过路由前缀（`/student/*`、`/tutor/*`）与本地登录态（token + userType）做分流与鉴权。
- 管理端为独立 SPA（`ai-tutor-admin-web`），独立登录态与路由鉴权。
- 需求发布者目前主要是学生；教师侧流程为：教师申请 → 发布者确认 → 教师支付信息费 → 开启聊天。

目标变化（见 proposal）：
- 引入“机构”作为新的需求发布方与收益方，且要求机构账号不能由 C 端自助创建；管理端创建并发放。
- 教师侧主流程尽量不变，但信息费收入需要按比例在平台与机构间归属，并支持退款/纠纷导致的冻结与冲正。

约束与干系人：
- 干系人：机构运营/客服、平台财务对账、管理员、学生/教师用户、第三方支付渠道。
- 约束：尽量复用现有前端工程与组件；在资金与鉴权上优先“安全正确”而非“最快上线”；所有资金变动可审计、可对账、可回滚（冲正）。

## Goals / Non-Goals

**Goals:**
- 前端不新建独立“机构端项目”，而是在 `ai-tutor-web` 内新增机构域路由与机构登录入口，最大化复用页面与组件，同时通过路由域与权限隔离做到体验与安全可控。
- 管理端保持在 `ai-tutor-admin-web` 内实现机构管理能力，与前台鉴权/令牌体系隔离。
- 后端将“机构发布单”抽象为与“学生发布单”同一生命周期的需求实体，教师侧申请/确认/付费/聊天闭环复用同一套流程与状态机。
- 引入信息费分账台账与结算控制：分账不是简单“支付成功即拆分不可逆”，而是以台账归属 + 可结算条件驱动，支持退款/纠纷冲正、冻结与审计。
- 鉴权与风控：机构账号仅可由管理端创建；机构登录入口与学生/教师登录入口分离；支持机构账号禁用/冻结、强制改密、登录频控与异常告警。

**Non-Goals:**
- 不在首期强制实现复杂的机构子账号/多角色权限矩阵（可预留数据结构与扩展点）。
- 不在首期实现对公打款、自动开票、复杂税务能力（仅预留字段与流程钩子）。
- 不在首期改变教师侧核心业务流程与 UI 信息架构（仅增加发布者类型展示与机构主页入口）。

## Decisions

1) 前端工程形态：复用而非新建
- 决策：不创建新的前端项目；机构端作为 `ai-tutor-web` 内的“第三业务域”，新增 `/org/*` 路由前缀与独立登录入口 `/auth/org`。
- 理由：
  - 现有工程已支持多角色分流（学生/教师），有成熟的 token 注入与路由守卫，可扩展到机构角色，复用成本最低。
  - 独立登录入口可满足“与学生用户隔离、避免自助注册机构账号”的产品约束，同时仍可复用底层鉴权与 UI 基础设施。
- 备选方案：
  - 新建机构端 SPA：隔离性更强，但会引入重复的路由/鉴权/HTTP 封装与发布成本；与当前“同工程多角色”风格不一致，首期不选。

2) 身份与权限模型：新增 ORG 身份 + 服务端强约束
- 决策：在用户体系中引入机构身份（例如 `userType=ORG`），并在服务端对关键接口做“发布者类型/归属”校验；机构账号无自助注册入口，仅允许管理端创建。
- 理由：
  - 前端路由守卫只能作为体验层限制，必须依赖服务端鉴权来防止越权（例如学生伪造请求创建机构单、机构越权查看其他机构对账）。
  - 与教师/学生一致的 token / session 机制有利于复用，差异在于权限与可访问资源集。
- 备选方案：
  - 完全独立机构账号体系与 token：隔离更强但实现成本更高，且会重复一套鉴权/会话/用户中心能力；可在后续增长后再演进。

3) 登录方式：机构账号采用“受控凭证”登录
- 决策：机构登录入口与学生/教师登录入口分离；机构账号使用管理端发放的登录凭证（建议：账号+密码，支持首次登录强制改密；必要时支持客服重置与短期临时密码）。
- 理由：
  - 与“避免个人学生用户创建机构账号”的目标一致；短信验证码更偏向开放注册，不利于账号治理。
  - 密码 + 强制改密 + 重置流程更利于机构多人交接的真实场景。

4) 订单与资金：以“台账归属 + 可结算条件”实现分账
- 决策：信息费支付成功后统一暂存在平台侧，不直接对机构结算；先写入“分账台账”（平台份额/机构份额），并在满足结算条件后再进入“可结算/已结算”状态。结算条件首期以“聊天内合作提案发起并通过（ACCEPTED）”作为成单节点；退款/纠纷通过“冻结 + 冲正”在台账层闭环。
- 理由：
  - 退款、拒付、纠纷在真实场景必然发生；若先分出去再追讨，风险与成本高。
  - 台账驱动可实现可审计、可对账、可追踪的资金闭环，也便于引入提现/结算周期与风控。
- 备选方案：
  - 调用支付渠道的实时分账能力：实现复杂、受渠道约束且对异常处理更脆弱；可作为后续优化。

5) 退款/纠纷：定义状态机与客服介入点
- 决策：为“信息费”定义最小可落地的退款/纠纷状态机（申请、受理、举证、仲裁、退款、拒绝、关闭），并提供管理员/客服的强制操作能力（冻结、冲正、强制退款/拒绝、封禁机构）。
- 理由：
  - 机构不是学生本人，沟通与履约不确定性更高；必须通过流程与权限兜底，避免口径混乱与资金不可控。

## Risks / Trade-offs

- [机构身份与既有 userType/路由前缀耦合] → 通过路由 meta + 角色白名单逐步替代“前缀硬编码”，并在服务端做强校验，前端仅做体验引导。
- [退款/纠纷口径不清导致大量客服成本] → 在 specs 中明确可执行的规则（时间窗、证据要求、默认判责、自动化超时处理），并在管理端沉淀 SOP 与操作面板。
- [台账口径与财务对账不一致] → 统一定义“支付订单/信息费订单/分账台账/结算单”的主键关系与状态含义，所有资金变动可追溯到原始支付流水与操作人。
- [机构违规/失联导致平台信用风险] → 上线机构风控策略（冻结结算、保证金预留、限额、黑名单），并将机构主页与发布单展示加入合规/风险提示与投诉入口。

## Migration Plan

- 第 0 阶段（最小闭环）：机构账号（管理端创建）+ 机构发布需求 + 教师侧完整闭环 + 信息费分账台账（仅记账不提现）+ 管理端查看机构单与台账。
- 第 1 阶段（风险可控后）：结算与提现（周期/阈值/冻结）+ 退款/纠纷流程 + 更完整的机构资料与资质审核。
- 回滚策略：机构域功能可通过 feature flag 关闭；资金层以台账冲正为主，避免物理删单；管理端保留审计日志以支持追责。

## Open Questions

- 首期不支持机构子账号，是否允许机构内部共享主账号的操作审计粒度到“账号级”即可？
- 信息费“结算条件”确认：以“合作提案通过（ACCEPTED）”作为成单节点后，是否还需要冷静期（例如 T+N 天未退款）再进入可结算？
- 退款时间窗与判责：在“发布者确认后但教师未支付/已支付未聊天/已聊天”不同阶段的退款策略与默认判责口径。
- 机构主页展示的合规信息范围（资质、营业执照、客服电话、地址）与隐私边界。
