## ADDED Requirements

### Requirement: 基于 Chrome 的 UI 自动化执行能力
系统 MUST 提供基于 Playwright 的 UI 自动化能力，并以 Chrome 浏览器通道作为默认执行目标，覆盖主流程的关键页面与用户旅程。

#### Scenario: 使用 Chrome 通道执行 smoke UI 用例
- **WHEN** 执行 UI smoke 用例集合
- **THEN** 测试运行器 MUST 使用 Chrome 浏览器通道启动并完成用例执行

### Requirement: UI 自动化覆盖三条主流程用户旅程
系统 MUST 提供端到端 UI 用例覆盖以下用户旅程：登录+资料+上传、发需求到 IM 沟通闭环、支付收银台轮询到解锁闭环，并在关键节点进行业务断言。

#### Scenario: 登录并进入“我的资料”完成头像上传
- **WHEN** 用户在前端完成登录并进入资料页上传头像
- **THEN** 页面 MUST 展示新的头像且刷新后仍保持一致

#### Scenario: 从需求入口发起沟通并在聊天页收发消息
- **WHEN** 教师在需求页面点击“立即沟通”进入聊天页并发送消息
- **THEN** 对端用户 MUST 在不刷新页面的情况下收到新消息并渲染

#### Scenario: 打开收银台并在支付成功后展示成功状态
- **WHEN** 用户通过业务入口新开收银台页面并进入轮询查单状态
- **THEN** 在支付单变为 SUCCESS 后页面 MUST 展示成功状态并停止轮询

### Requirement: 采用可维护的页面对象模型（POM）与稳定性策略
系统 MUST 使用页面对象模型（POM）或等价抽象封装页面操作，并提供统一的稳定性策略，包括但不限于：明确等待策略、可观测事件断言、失败截图与追踪信息采集。

#### Scenario: UI 用例仅表达业务意图而不直接耦合低层选择器
- **WHEN** 编写 UI 用例以执行登录、导航、发起沟通、支付等操作
- **THEN** 用例 MUST 通过页面对象/组件对象调用完成关键动作，而非在用例中堆叠大量选择器与细节交互

### Requirement: UI 失败定位产物输出
系统 MUST 在 UI 用例失败时输出可用于排障的产物（至少包含截图；可选包含 trace、控制台日志与网络请求摘要），并与用例名称关联以便在 CI 归档。

#### Scenario: UI 用例失败自动产出截图
- **WHEN** 任意 UI 用例执行失败
- **THEN** 测试框架 MUST 保存失败截图并可在报告或 CI 产物中定位到对应截图文件
