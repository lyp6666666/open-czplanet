## ADDED Requirements

### Requirement: 测试环境可一键准备并具备健康检查
系统 MUST 提供测试环境准备方式，使得本地与 CI 可在执行测试前启动并验证依赖服务就绪（至少包含：MySQL、Redis、MinIO；可选包含 MQ）。

#### Scenario: 执行自动化前完成依赖服务健康检查
- **WHEN** 执行 smoke/regression 测试前运行环境准备流程
- **THEN** 测试框架 MUST 校验后端服务可达且依赖服务处于就绪状态，否则 MUST 明确失败并输出原因

### Requirement: 复用稳定的初始化数据作为基线
系统 MUST 复用仓库内提供的初始化 SQL 或等价数据装配机制，保证关键用户、教师档案与需求数据在测试启动后可稳定存在。

#### Scenario: 用例可使用初始化数据定位到可沟通的需求
- **WHEN** UI/API 用例需要从“需求列表”选择一条需求发起沟通
- **THEN** 测试数据基线 MUST 提供至少一条可用需求数据或可通过 API 创建并成功展示

### Requirement: 测试数据创建必须幂等且可清理
系统 MUST 提供数据创建与清理机制，保证重复执行用例不会因脏数据导致失败，并避免用例之间相互污染。

#### Scenario: 重复执行同一用例不产生不可控的重复数据
- **WHEN** 连续两次执行同一条创建类用例（例如创建需求/发起支付/发送消息）
- **THEN** 系统 MUST 通过幂等键、唯一标识或 teardown 清理策略保证第二次执行可通过且数据可追踪

### Requirement: 登录验证码在测试环境必须可获取且不依赖外部短信
系统 MUST 提供仅在测试环境启用的验证码获取策略，使自动化测试可以完成登录而不依赖外部短信平台或真实短信配置。

#### Scenario: 测试用例可在不配置短信 token 的情况下完成登录
- **WHEN** 测试环境未配置外部短信 token 或外网不可达
- **THEN** 自动化测试 MUST 仍可获得验证码并完成 `/user/loginOrRegister` 登录流程

### Requirement: 支付回调在测试环境必须可被模拟触发
系统 MUST 支持在测试环境中模拟第三方支付回调，触发支付单状态迁移与业务解锁链路，用于闭环 E2E 验证。

#### Scenario: 测试侧可构造签名参数触发回调成功
- **WHEN** 测试用例构造符合验签规则的回调参数并调用 `/payment/notify/yungouos`
- **THEN** 支付单状态 MUST 迁移为 SUCCESS 且业务侧可观测到解锁结果
