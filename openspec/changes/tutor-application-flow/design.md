## Context

当前教师可从需求页直接进入 1v1 聊天与学员沟通，导致平台容易被绕过。目标是将核心链路收口为：发起申请 → 对方通过 → 教师支付中介费 → 开启聊天。申请既支持教师向学员发起，也支持学员向教员发起，但中介费付款方始终为教师。

本变更涉及：
- 需求页/教员页入口形态变更（聊天入口改为申请入口）
- 新增申请中心（发出/收到、未读、处理）
- 支付与聊天解锁的强制后端拦截（避免仅前端限制被绕过）
- 与既有 IM 会话创建/进入能力、既有中介费订单/支付能力的衔接

约束与原则：
- 聊天解锁必须由后端强约束，任何入口（需求页、消息页、房间直达）都不可绕过
- 关键状态均可追溯（申请、订单、解锁）
- 新流程需兼容教师/学员双向发起申请，但支付责任固定为教师

## Goals / Non-Goals

**Goals:**
- 引入“申请”作为聊天前置步骤，统一找家教流程并降低绕过风险
- 提供申请中心：可查看我发出的/我收到的申请，支持未读与处理（通过/拒绝）
- 在申请通过后，强制教师完成中介费支付才允许双方进入聊天
- 支付完成后创建或复用 1v1 room，并将用户导入聊天

**Non-Goals:**
- 不在本变更中定义“合作提案/展示联系方式”的完整产品与支付策略，只保证聊天前置链路可闭环
- 不强制替换既有中介费订单与支付方案实现细节（沿用现有订单/支付能力，按需扩展绑定关系）
- 不在本变更中改造所有历史会话的访问策略，仅对“与申请绑定的新链路”强制 gating

## Decisions

### Decision: 以“申请”为业务主键，统一关联支付与聊天解锁
选择以 applicationId 作为主线对象，申请通过后生成/绑定中介费订单，支付完成后解锁聊天并关联 roomId。

原因：
- 可将“沟通入口”从房间创建迁移为申请创建，使链路可审计、可风控
- 支付与聊天解锁能通过 applicationId 做一一绑定与幂等

备选：
- 以 demandId/tutorId 直接绑定订单与聊天解锁：难以处理多次沟通、多次申请与审计追溯

### Decision: 聊天进入统一后端拦截，返回可操作的 gating 状态
后端在“进入/创建 chat room”路径统一检查 applicationId 的 chatAccessStatus，并返回明确的 gating 响应：
- 对教师：paymentRequired + orderId（引导去支付）
- 对学员：waitingForTeacherPayment（告知等待教师支付）

原因：
- 避免仅靠前端路由或按钮禁用造成绕过
- 便于客户端统一处理不同入口（申请中心、消息列表、推送直达）

备选：
- 仅在申请中心入口做拦截：仍可从消息页或 room 直达绕过

### Decision: 支付责任固定为教师，订单 payerRole 强制为 TEACHER
无论申请由谁发起，申请通过后生成/获取订单时，均由后端强制设置 payerRole = TEACHER，并以教师身份做权限校验与支付流程。

原因：
- 与业务策略一致，避免客户端传参导致的支付主体错乱
- 支持“学员发起申请”的场景而不改变结算逻辑

备选：
- 按发起方决定付款人：与业务策略冲突，会引入复杂分支

### Decision: 幂等与状态机在核心写路径强制实现
对申请创建、申请处理（通过/拒绝）、订单创建/获取、room 创建/获取均要求幂等与受控状态机。

原因：
- 客户端重试、网络抖动、并发操作是常态，必须保证不会生成重复申请/订单/房间
- 状态机受控便于后续风控与运营统计

## Risks / Trade-offs

- [风险] 与既有“需求入口发起沟通”能力冲突（旧入口仍可创建 room） → [缓解] 将旧入口改为发起申请；并在后端对与申请绑定的进入接口强制校验；逐步下线旧“直接沟通”能力或在其上增加 gating 开关
- [风险] 历史 room 或外部 deep link 仍可能绕过 → [缓解] 将“进入聊天”改造为必须携带 applicationId 或可从 roomId 反查绑定的 applicationId 并校验 gating；对未绑定申请的历史 room 维持旧策略但不在新入口导流
- [风险] 支付确认异步导致解锁延迟 → [缓解] 订单状态变更为 `PAID` 时驱动解锁；客户端轮询/订阅订单状态并刷新申请详情
- [风险] 未读与列表一致性复杂（多端登录） → [缓解] 未读状态以服务端为准，读状态变更由“查看详情”触发，未读数接口提供最终一致读模型

## Migration Plan

1. 引入申请数据模型与基础接口（创建、列表、详情、通过/拒绝、未读）
2. 实现“申请通过 → 创建/获取订单（付款人为教师）”的绑定逻辑
3. 在 chat 进入/创建路径增加 gating 拦截，返回 paymentRequired / waitingForTeacherPayment / roomId
4. 前端将需求页/教员页“发起聊天”改为“发起申请”，增加申请编写弹层
5. 新增申请中心页面，接入未读提示与处理流转
6. 灰度开关：按端/按人群逐步切流；监控申请创建量、通过率、支付转化、进入聊天成功率
7. 回滚策略：保留旧直接聊天入口的开关；若出现重大问题可临时放开 gating，同时保留申请能力不影响数据

## Open Questions

- 申请与 chat room 的绑定方式：进入聊天是否强制携带 applicationId，还是允许从 roomId 反查绑定关系
- 订单能力复用范围：是否直接复用既有“合作提案生成中介费订单”能力，还是新增“按申请生成订单”接口并在内部复用订单域逻辑
- 申请内容的审核与敏感词策略是否需要在本次上线前纳入（例如联系方式拦截）
